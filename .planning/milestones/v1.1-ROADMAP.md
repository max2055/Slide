## Phase Details

### Phase 84: RBAC Foundation
**Goal**: Admin can manage custom roles, permission points, user-role bindings, and instance-level access through backend APIs, with middleware enforcing access control on all protected routes
**Depends on**: Nothing (first milestone phase)
**Requirements**: RBAC-01, RBAC-02, RBAC-03, RBAC-04, RBAC-05, RBAC-06, RBAC-07, RBAC-08
**Success Criteria** (what must be TRUE):
  1. Admin can create, edit, and delete custom roles with name and description via API
  2. Admin can define permission codes following `resource:action` namespace convention and assign them to roles
  3. Admin can assign multiple roles to a user (many-to-many) and grant instance-level access
  4. `requirePermission` and `requireInstanceAccess` Fastify preHandler middlewares enforce access on all protected routes, returning 403 for unauthorized requests
  5. Existing users are migrated from `users.role` ENUM to `user_roles` table without any permission loss
**Plans**: 4 plans (Wave 1: 1 plan, Wave 2: 1 plan, Wave 3: 1 plan, Wave 4: 1 gap closure plan)

Plans:
- [x] 84-01-PLAN.md -- RBAC Data Layer: SQL migration (5 tables + seed + user migration), RbacService class, unit tests
- [x] 84-02-PLAN.md -- Middleware + API: requirePermission, requireInstanceAccess factories, RBAC management API at /api/v1/rbac/*
- [x] 84-03-PLAN.md -- Route Integration + Cleanup: update 139 route registrations, register RBAC API, delete old requireRole, update auth-database-service
- [x] 84-04-PLAN.md -- Gap Closure: Fix CR-01 (verifyToken decoration), CR-02 (role column references), CR-03 (wildcard check), WR-03 (collector permissions)

### Phase 86: SQL Console Upgrade
**Goal**: SQL editor provides schema-driven autocomplete with multi-tab support, result table with sort/pagination/CSV export, query history, and EXPLAIN visualization
**Depends on**: Nothing (can run in parallel with Phase 85, 87)
**Requirements**: SQLC-01, SQLC-02, SQLC-03, SQLC-04, SQLC-05, SQLC-06, SQLC-07
**Success Criteria** (what must be TRUE):
  1. SQL editor suggests table and column names as user types, sourced from live instance schema
  2. User can click column headers to sort results ascending/descending/none, and configure page size for client-side pagination
  3. User can export current query results to CSV via a download button
  4. User can open, switch, and close multiple query tabs, each with independent editor state and result set
  5. Query history is automatically saved (SQL text, instance, duration, row count) and searchable from a history panel
  6. EXPLAIN JSON output is rendered as a visual tree/table component in the console
**Plans**: 5 plans (Wave 1: 1 plan, Wave 2: 1 plan, Wave 3: 1 plan, Wave 4: 1 plan, Wave 5: 1 plan)
**UI hint**: yes

### Phase 87: Approval Enhancement
**Goal**: Approver can review SQL with syntax highlighting, see full approval timeline, batch-approve/reject, optionally auto-execute SQL, and receive notifications
**Depends on**: Nothing (can run in parallel with Phase 85, 86)
**Requirements**: APPR-01, APPR-02, APPR-03, APPR-04, APPR-05, APPR-06
**Success Criteria** (what must be TRUE):
  1. Approval detail page shows SQL with syntax highlighting in a read-only CodeMirror editor and full history timeline (submit/review/execute/notify events)
  2. Approver can batch-select multiple pending requests and approve/reject all in one action with a single review note
  3. Approve action can optionally auto-execute the SQL, recording execution result (success/failure, rows affected, duration) on the approval record
  4. Reject action writes the reviewer comment back to the approval request
  5. Approval outcome notifies the requester via configured notification channels (DingTalk/WeCom/Feishu/Webhook)
**Plans**: 4 plans (Wave 1: 1 plan, Wave 2: 1 plan, Wave 3: 1 plan, Wave 4: 1 plan)
**UI hint**: yes

Plans:
- [x] 87-01-PLAN.md -- Backend Data + Service: approval_events table, event CRUD, execute_after_approve, batchReview
- [x] 87-02-PLAN.md -- Backend API + Notifications: buildApprovalMessage, batch-review endpoint, events endpoint, notification integration
- [x] 87-03-PLAN.md -- Frontend List + Batch: sub-view switching, checkbox multi-select, batch bar, execute toggle, batch dialog

  6. EXPLAIN normalizer handles MySQL 5.7 flat format, MariaDB variants, and PostgreSQL planner-extended output
