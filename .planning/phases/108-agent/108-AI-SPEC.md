## 1. System Classification

**System Type:** Hybrid (Multi-Agent + Autonomous Task Agent + Conversational)

**Description:**
Phase 108 defines the `IAgentEngine` interface contract to decouple Slide's platform code from the underlying Agent engine. The system has three adapters: `DirectAdapter` (default, using `@slide/agent-core` — a custom TypeScript agent engine ported from nanobot's proven patterns), `OpenClawAdapter` (legacy compatibility, wrapping existing OpenClaw WebSocket/RPC code), and `nanobotAdapter` (Phase 109, HTTP to Python process as optional fallback). Platform code only depends on the interface, not any specific adapter. "Good" means: Chat, AI analysis, and tool execution work identically regardless of which adapter is active, with no OpenClaw imports in platform code.

**Critical Failure Modes:**
1. Platform code breaks on adapter switch — users can't send Chat messages or receive streaming responses after toggling adapters
2. Tool registration mismatch — tools work in one adapter but silently fail or return wrong results in another
3. AI analysis regression — fire-and-forget `invoke()` tasks fail, causing alert analysis gaps
4. Streaming fidelity loss — text_delta, tool_start, tool_result events missing or malformed across adapters
5. Dead code removal breaks live features — deleting agent-service.ts exports that are still imported by server.ts or chat-methods.ts

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer Tooling — Database Administration (DBA) Platform

**User Population:** DBAs, database reliability engineers, SREs, and backend developers managing MySQL, Oracle, Dameng, MongoDB, and Elasticsearch instances in enterprise environments. Users interact through a chat interface: asking about database health, investigating slow queries, reviewing alerts, and requesting diagnostic tool execution. English and Chinese language users.

**Stakes Level:** High

**Output Consequence:** AI output is consumed by DBAs who act on it — running suggested queries against production databases, applying diagnostic conclusions to incident response, and trusting health-check results for capacity planning. Incorrect analysis can misdirect root-cause investigation, cause unnecessary downtime during fix attempts, or mask real issues. The abstraction layer (Phase 108) does not change what the AI says, but if it breaks silently, DBAs lose trust in every downstream answer.

### What Domain Experts Evaluate Against

Phase 108 is a platform infrastructure refactoring — it introduces an adapter layer with zero user-facing behavioral change. Domain experts evaluate it by verifying that existing DBA workflows survive the refactoring intact. The rubric below is framed for the adapter layer behavior, not the AI output quality (which is unchanged).

```
Dimension: Tool registration parity across adapters
Good: All platform tools registered in catalog.ts (db_health_check, db_slow_queries, db_explain_query, etc.) are available with identical name, description, and JSON Schema parameters regardless of which adapter is active. A DBA asking "check health of instance-01" gets the same tool call triggered.
Bad: A tool is registered in DirectAdapter but missing or differently parameterized in OpenClawAdapter. The DBA sees "unknown tool" or a silent no-op. Or a parameter that works in one adapter (e.g., instanceId as string) fails schema validation in another.
Stakes: Critical
Source: Practitioner knowledge — tooling inconsistency is the #1 cause of silent production failures in AI agent deployments (DAB benchmark, arXiv 2603.20576; BIRD-INTERACT ICLR 2026)

Dimension: Streaming event fidelity during chat
Good: Chat produces the identical sequence of typed ChatEvent payloads (text_delta, tool_start, tool_result, complete) for the same user input regardless of adapter. DBAs see tool calls appearing and completing at the same points in the stream. No events dropped, duplicated, or reordered.
Bad: The OpenClawAdapter emits tool_start before the first text_delta while DirectAdapter emits text_delta first. Or tool_result events are missing when switching to DirectAdapter. DBAs see the chat output jump or skip — a tool that was "obviously running" in the old adapter appears to hang in the new one.
Stakes: Critical
Source: Practitioner knowledge — streaming event loss is invisible to code-level checks but immediately noticed by human users (Nearform, 2026; Label Studio domain expert best practices)

Dimension: AI analysis (invoke) reliability
Good: Fire-and-forget invoke() calls (alert analysis, metric diagnosis) complete with consistent content, structure, and confidence for the same input regardless of adapter. Analysis results are persisted and surfaced in the alert detail view identically.
Bad: An alert triggers, the analysis adapter runs, but invoke() returns empty, errors silently, or the adapter misformats the result. The alert remains unanalyzed in the dashboard. The DBA sees "pending" indefinitely with no error signal.
Stakes: High
Source: Practitioner knowledge — silent analysis failures are the most-reported pain point in DBA agent deployments (SCALE 2.0 evaluation framework, 2025; DBA Agent research, emergentmind.com 2026)

Dimension: Session history continuity across adapter switches
Good: A DBA having a multi-turn conversation about a production incident can switch adapters (e.g., toggle flag OpenClawAdapter -> DirectAdapter) and continue the conversation without losing context. Previous assistant responses are preserved in the chat history view.
Bad: Adapter switch drops the in-memory session state, and the AI either has no memory of previous turns or the history view shows broken/partial messages. The DBA must re-explain the incident context.
Stakes: High
Source: Practitioner knowledge — session continuity is a non-negotiable UX requirement for enterprise tooling (BIRD-INTERACT benchmark, ICLR 2026; Phase 108 D-14/D-15 design decisions)

Dimension: Error transparency for adapter-level failures
Good: When an adapter cannot process a request (model unavailable, WebSocket disconnected, tool handler crash), the error is surfaced to the platform with the same structure and detail across all adapters. The DBA sees an actionable error message, not a silent hang or a confusing generic failure.
Bad: DirectAdapter throws a typed Error that the platform catches and displays as "Tool execution timed out". OpenClawAdapter swallows the same error and returns an empty success. The DBA has no way to distinguish "analysis completed with no findings" from "analysis never ran."
Stakes: High
Source: Practitioner knowledge — silent error swallowing is the root cause of the "governance gap" identified by the Liquibase 2026 State of Database Change Governance Report
```

### Known Failure Modes in This Domain

These are failure modes specific to adapter layer behavior in a DBA tool context, not generic LLM failure modes.

1. **Silent tool mismatch.** The OpenClawAdapter inherits pi-agent base tools (bash, read, write) internally. The DirectAdapter does not. If platform code relies on tool discovery via `listTools()` and a consumer expects bash access, the adapter switch silently removes capability. Mitigation: `capabilities()` must accurately report tool availability, and platform code must check it before dispatching.

2. **Streaming event translation loss.** The OpenClaw WebSocket protocol emits events in a different format than `@slide/agent-core` emits internally. The adapter layer translates between them. If the translation misses an event type (e.g., OpenClaw's `"agent:thinking"` has no equivalent in ChatEvent), the frontend either hangs or shows incomplete output. This is the most common failure in production adapter layers (Nearform 2026).

3. **Fire-and-forget blind spot.** The `invoke()` method is used for AI analysis of database alerts — a critical path. If invoke returns a rejected promise (adapter crash) but the platform caller uses `.catch()` incorrectly or the adapter returns success with empty content, no alert analysis is generated and the gap is invisible. The 2026 DAB benchmark found this pattern accounts for ~15% of production failures in data-oriented agents.

4. **Database tool read-only contract violation.** Tools like `db_health_check` are marked `readOnly: true` and `concurrencySafe: true` in DirectAdapter. If the OpenClawAdapter wraps a tool that connects to the database in read-write mode (e.g., because the underlying OpenClaw runtime does not enforce read-only flags), the abstraction layer silently permits a DBA tool to modify production data. This violates the separation of concerns defined in D-11 (pi-agent tools belong in OpenClaw only; platform tools are shared).

5. **Context window budget divergence.** DirectAdapter uses `@slide/agent-core` context governance (microcompact + snip) with a hard budget of 200K tokens. OpenClawAdapter uses the LLM provider's default context management. For the same multi-turn conversation, one adapter may truncate history at 50K tokens while the other keeps 180K — producing different LLM behavior on the same input. DBAs doing long incident investigations will see inconsistent recall depending on which adapter is active.

### Regulatory / Compliance Context

No external financial or healthcare regulations apply directly to this abstraction layer. However, enterprise database operations impose internal compliance constraints that the adapter must preserve:

- **Database credential isolation (internal standard).** Adapters must not expose database credentials, hostnames, or connection strings in tool parameters or streaming events visible to the frontend. The existing `catalog.ts` tools accept `instanceId` (not raw credentials); this contract must hold across all adapters.
- **Audit trail continuity (internal standard, per REQUIREMENTS.md SEC-01 through SEC-04 and AUTH-01 through AUTH-05).** All AI analysis results must be logged with the originating adapter identifier so that post-incident reviews can trace which engine produced a given recommendation. The Liquibase 2026 report identifies this as the top governance gap in AI-database tooling: "ungoverned AI-generated operations" (46.5% of enterprises cite this as a key concern).
- **Least-privilege preservation (internal standard).** The adapter layer must not bypass the existing instance-level access controls (read-only / read-write / admin per instance). OpenClawAdapter's internal pi-agent tools (bash, read, write) must remain walled off from the platform tool path.
- **No direct SQL execution by LLM (industry best practice).** The adapter layer is a pass-through for tool execution, not a SQL console. All database interaction goes through deterministic, validated tool handlers. This boundary is explicitly maintained by the `IAgentEngine` interface (tools are registered by the platform, not by the adapter).
- **SM series encryption compliance (Dameng scenarios).** Slide supports Dameng database instances common in Chinese enterprise deployments that require national cryptography standards (SM2/SM3/SM4). The adapter layer must not interfere with any TLS or encryption configuration already in place at the database connection level.

## 2. Framework Decision

**Selected Framework:** @slide/agent-core v0.1.0 (custom in-house TypeScript agent engine)

**Version:** v0.1.0 (workspace package in `packages/agent-core/`)

**Rationale:**
Phase 108 defines the `IAgentEngine` interface around an already-built custom engine. `@slide/agent-core` (1523 lines TS, compiled, zero external AI framework dependencies) provides AgentRunner with parallel tool execution, checkpointing, context budget management (microcompact + snip), and structured tracing — ported from nanobot's proven Python patterns to TypeScript. Using a custom engine avoids abstraction overhead, vendor lock-in, and Python-only limitations of off-the-shelf frameworks while keeping the entire stack in TypeScript within the existing monorepo.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain/LangGraph | Python-only for core features; abstraction overhead; overkill for Slide's focused use case |
| CrewAI | Python only; multi-agent orchestration is not Slide's primary need |
| OpenAI Agents SDK | Vendor lock-in; Slide needs Anthropic + Ollama support |
| Claude Agent SDK | Claude-only; Slide needs model-agnostic provider layer |
| Direct SDK (no engine) | Lacks checkpointing, context compaction, and structured tracing that agent-core provides |

**Vendor Lock-In Accepted:** No — `@slide/agent-core` wraps Anthropic SDK, OpenAI SDK, and Ollama behind a unified `LLMProvider` interface. Model-agnostic by design.

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `AgentRunner` | LLM-to-tool execution loop. Manages multi-turn conversation: send to LLM, parse tool calls, execute tools, feed results back, repeat. Contains all 6 key mechanisms (parallel tools, injections, checkpointing, context budget, timeout, tracing). | Every time you need an agent that can use tools. The core entry point for chat, analysis, and autonomous tasks. |
| `ToolRegistry` | Dynamic tool registration + JSON Schema validation + execution dispatch. Ported from nanobot `registry.py`. Caches tool definitions; rejects invalid params before they reach the handler. | One registry per agent instance. Tools are registered once at startup (D-13). |
| `AgentHook` (interface) | Lifecycle hooks that customise the runner's behaviour without modifying the loop. `beforeIteration`, `onStream`, `onStreamEnd`, `beforeExecuteTools`, `afterIteration`, `finalizeContent`. | Implement for streaming output, checkpointing, tracing, tool event capture. The `NoopHook` is the default for non-streaming use. |
| `LLMProvider` (interface) | Model-agnostic abstraction over Anthropic SDK, OpenAI SDK, Ollama, etc. Must implement `chat` (non-streaming) and `chatStream`. | Wrap any LLM provider behind a single interface. Switch providers without changing agent logic. |
| `AgentRunSpec` | Complete configuration for one agent run: messages, tools, model, iteration limit, context budget, timeout, injection callback, checkpoint callback. | Pass to `runner.run()`. Defines everything the runner needs for one execution. |
| Context Governance (internal) | `microcompact` + `snip` + `backfillMissingToolResults` + `dropOrphanToolResults`. Built-in context window management that compacts tool results and truncates history when budget is exceeded. | Automatic — no developer action needed, but understand the mechanism to tune `contextWindowTokens` and `maxToolResultChars`. |

### Common Pitfalls

1. **Not implementing `chatStream` when the hook wants streaming.** If `hook.wantsStreaming()` returns `true` but the `LLMProvider` does not implement `chatStream` (or the implementation does not call `callbacks.onContentDelta`), the runner will receive no stream deltas and stream-observing hooks will show no output. Always pair streaming providers with streaming hooks.

2. **Tool concurrency safety misconfiguration.** Tools with `concurrencySafe: false` are never batched in parallel execution (`partitionToolBatches`). If a tool mutates shared state and is incorrectly marked `concurrencySafe: true`, parallel execution can cause race conditions. Default to `false` for any tool with side effects; only set `true` for pure read tools.

3. **Context budget set too low for large tool results.** The default `maxToolResultChars` is 20000 (`spec.maxToolResultChars`). If tools return large JSON payloads (e.g., database query results), the snip mechanism will aggressively truncate history, potentially losing earlier user instructions. Tune `contextWindowTokens` and `maxToolResultChars` together, not independently.

4. **Infinite loop risk from repeated tool calls.** The runner has built-in guards in `repeatedExternalLookupError` (5+ calls on same tool) and `repeatedWorkspaceViolationError` (3+ violations), but the guard checks are per-tool-name counters. A tool that changes its `name` each invocation would bypass the guard. If you wrap tools with dynamic names, add your own loop detection in the `AgentHook.beforeExecuteTools` callback.

5. **Mid-turn injection drains across iteration cycles.** The `injectionCallback` is called after every tool execution and final response (up to `MAX_INJECTION_CYCLES = 5`). If the injection callback returns new user messages on every call (e.g., from an external event queue that produces messages faster than the agent consumes them), the runner can enter an injection-drain loop that delays the final response. Implement backpressure in your injection callback (e.g., return empty array if too many consecutive injections have been processed).

6. **Zero-copy assumption on `AgentRunner.setProvider()`.** The `setProvider()` method replaces the runner's provider, but any *in-flight* `run()` call continues using the old provider because the closure was already created. Always set the provider before calling `run()`, not concurrently.

## 4. Implementation Guidance

**Model Configuration:**

The Hybrid system uses three model tiers depending on the operation:

| Operation | Model | Temperature | Max Tokens | Reasoning Effort |
|-----------|-------|-------------|------------|------------------|
| Chat (streaming) | `claude-sonnet-4-20250929` (primary) / `gpt-4o` (secondary) | 0.0 | 4096 | — |
| AI analysis (invoke) | `claude-sonnet-4-20250929` | 0.0 | 2048 | — |
| Tool-intensive tasks | `claude-sonnet-4-20250929` | 0.0 | 8192 | — |
| Classification/routing | `claude-haiku-3` or `gpt-4o-mini` | 0.0 | 512 | — |

Configured via `LLMCallOptions` in the `LLMProvider` implementation. The provider reads model and key from `.env`:
```
ANTHROPIC_MODEL=claude-sonnet-4-20250929
OPENAI_MODEL=gpt-4o
```

**Core Pattern:**

The `IAgentEngine` interface defines two execution paths that map to `AgentRunner.run()` with different configurations:

```typescript
// IAgentEngine interface (defined in apps/db-ops-api/src/adapter/types.ts)
interface IAgentEngine {
  /** Streaming chat session. onEvent receives typed ChatEvent payloads. */
  chat(
    sessionKey: string,
    message: string,
    onEvent: (event: ChatEvent) => void,
  ): Promise<ChatResult>;

  /** Fire-and-forget task execution (AI analysis, alerts, etc.). */
  invoke(
    sessionKey: string,
    message: string,
    systemPrompt?: string,
  ): Promise<InvokeResult>;

  /** List all registered tools with their schemas. */
  listTools(): ToolSchema[];

  /** Query adapter capabilities for graceful degradation. */
  capabilities(): AgentCapabilities;
}

// DirectAdapter wraps AgentRunner.run() behind IAgentEngine
class DirectAdapter implements IAgentEngine {
  private runner: AgentRunner;
  private registry: ToolRegistry;
  private provider: LLMProvider;
  private sessions: Map<string, Message[]> = new Map();

  async chat(
    sessionKey: string,
    message: string,
    onEvent: (event: ChatEvent) => void,
  ): Promise<ChatResult> {
    // 1. Load or create session history
    let history = this.sessions.get(sessionKey);
    if (!history) {
      history = [{ role: "system", content: SYSTEM_PROMPT }];
      this.sessions.set(sessionKey, history);
    }
    history.push({ role: "user", content: message });

    // 2. Configure streaming hook that dispatches onEvent
    const hook = new ChatStreamHook(onEvent);

    // 3. Delegate to AgentRunner
    const result = await this.runner.run({
      initialMessages: history,
      tools: this.registry,
      model: this.provider.getDefaultModel(),
      maxIterations: 10,
      maxToolResultChars: 20000,
      temperature: 0.0,
      hook,
      contextWindowTokens: 200_000,
      maxTokens: 4096,
    });

    // 4. Persist result to session history
    history.push({ role: "assistant", content: result.finalContent });
    return { finalContent: result.finalContent, usage: result.usage };
  }

  async invoke(
    sessionKey: string,
    message: string,
    systemPrompt?: string,
  ): Promise<InvokeResult> {
    // Fire-and-forget: non-streaming, no session persistence
    const messages: Message[] = [
      { role: "system", content: systemPrompt ?? DEFAULT_INVOKE_PROMPT },
      { role: "user", content: message },
    ];

    const result = await this.runner.run({
      initialMessages: messages,
      tools: this.registry,
      model: this.provider.getDefaultModel(),
      maxIterations: 5,
      maxToolResultChars: 20000,
      temperature: 0.0,
      hook: new NoopHook(),
      contextWindowTokens: 200_000,
      maxTokens: 2048,
    });

    return { content: result.finalContent, usage: result.usage };
  }
}
```

**Tool Use:**

Tools follow a two-tier architecture:

1. **Platform tools** (Slide-owned): Database operations (db_health_check, db_slow_queries, etc.) defined in `apps/db-ops-api/src/tools/catalog.ts`. Handlers use closure-injected service instances. Registered on the `ToolRegistry`.

```typescript
// In catalog.ts
import { type Tool } from "@slide/agent-core";

export function createPlatformTools(dbService: DatabaseService): Tool[] {
  return [
    {
      name: "db_health_check",
      description: "Run health check against a database instance",
      parameters: {
        type: "object",
        properties: {
          instanceId: { type: "string", description: "Database instance ID" },
        },
        required: ["instanceId"],
      },
      readOnly: true,
      concurrencySafe: true,
      exclusive: false,
      async execute(params) {
        return dbService.healthCheck(params.instanceId as string);
      },
    },
  ];
}
```

2. **Adapter-specific tools**: OpenClaw adapter registers pi-agent base tools (bash, read, write, edit) internally. DirectAdapter does not include them — only platform tools are shared.

The `ToolRegistry.execute()` method wraps all calls with JSON Schema validation (`validateJsonSchema`) and parameter coercion (`castToolParams`). If validation fails, the error is returned as a string hint to the LLM — the LLM can then retry with corrected parameters.

**State Management:**

State is managed at three levels:

| Level | Mechanism | Scope |
|-------|-----------|-------|
| Session history | `Map<string, Message[]>` in DirectAdapter | Per-session chat state; persisted to MySQL via `chatDatabaseService` (platform responsibility, not adapter) |
| Agent run state | `AgentRunResult.messages` returned by `runner.run()` | Single run; includes all messages including tool results |
| Checkpointing | `checkpointCallback` in `AgentRunSpec` | Optional: write checkpoints to durable storage for interrupt recovery |

Key rule (D-14, D-15): The adapter does not own persistence. Session messages are returned to the platform which persists them via `chatDatabaseService`. The adapter only keeps an in-memory cache for the duration of the chat session.

- For `chat()`: history is loaded from `chatDatabaseService` before each run, and the new assistant message is saved after the run completes.
- For `invoke()`: no history is persisted. The messages are ephemeral.
- For `DirectAdapter`: the internal `sessions` Map is a cache. On process restart, history is rehydrated from MySQL.

**Context Window Strategy:**

The `AgentRunner` has built-in context governance that runs before every LLM call:

1. **`dropOrphanToolResults()`**: Removes tool result messages whose corresponding tool call was already snipped.
2. **`backfillMissingToolResults()`**: Inserts placeholder messages for tool calls that are missing results (e.g., from a previous run's checkpoint).
3. **`microcompact()`**: Compacts tool results from high-volume tools (read_file, exec, grep, find_files, web_search, web_fetch, list_dir). Keeps the most recent 10 detailed results; replaces older ones with `"[tool result omitted from context]"`.
4. **`applyToolResultBudget()`**: Truncates individual tool results exceeding `maxToolResultChars`.
5. **`snipHistory()`**: If estimated token count > `contextWindowTokens - maxTokens - 1024`, keeps only the most recent messages fitting within budget. System messages are preserved.

**Tuning for this system type:**
- Chat sessions: `contextWindowTokens: 200_000` (full Claude 4 context). After about 20-30 tool turns, `snipHistory` will start truncating.
- AI analysis: `contextWindowTokens: 100_000` with `maxIterations: 5`. Analysis tasks are short; snip rarely triggers.
- If `snipHistory` drops important context (e.g., the original user instruction), raise `contextWindowTokens` or reduce `maxToolResultChars` so more history fits.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Zod

For TypeScript/Node.js projects, Zod serves the same role as Pydantic in Python: define the output schema and have the LLM produce structured data that is validated before use.

**Pattern for this system type (Hybrid: chat + tool tasks):**

```typescript
import { z } from "zod";

// 1. Define the output schema with Zod
const AlertAnalysisSchema = z.object({
  severity: z.enum(["critical", "high", "medium", "low", "info"]),
  summary: z.string().min(10).max(500),
  rootCause: z.string().optional(),
  recommendedAction: z.string().min(1),
  affectedServices: z.array(z.string()).max(10),
  confidence: z.number().min(0).max(1),
});

type AlertAnalysis = z.infer<typeof AlertAnalysisSchema>;

// 2. Integrate with @slide/agent-core
// Since agent-core does not have a native structured output parser,
// use the system prompt to enforce JSON, then validate with Zod.
async function analyzeAlert(
  runner: AgentRunner,
  registry: ToolRegistry,
  alertText: string,
): Promise<AlertAnalysis> {
  const MAX_RETRIES = 2;

  for (let attempt = 0; attempt <= MAX_RETRIES; attempt++) {
    const result = await runner.run({
      initialMessages: [
        {
          role: "system",
          content: [
            "You are an alert analysis system. ",
            "You MUST respond with a valid JSON object matching this Zod schema:",
            JSON.stringify(AlertAnalysisSchema.shape, null, 2),
            "Do NOT include markdown code fences or any text outside the JSON object.",
          ].join("\n"),
        },
        { role: "user", content: `Analyze this alert: ${alertText}` },
      ],
      tools: registry,
      model: "claude-sonnet-4-20250929",
      maxIterations: 1,          // Single turn — no tool use for analysis
      maxToolResultChars: 0,     // No tool results expected
      temperature: 0.0,
      maxTokens: 1024,
      hook: new NoopHook(),
    });

    // 3. Validate with Zod
    const raw = result.finalContent?.trim() ?? "";
    // Strip markdown code fences if present
    const json = raw.replace(/^```(?:json)?\s*/, "").replace(/\s*```$/, "");

    try {
      const parsed = JSON.parse(json);
      const validated = AlertAnalysisSchema.parse(parsed);
      return validated;
    } catch (err) {
      if (attempt === MAX_RETRIES) {
        // Surface to monitoring — do not swallow in production
        console.error(`Structured output validation failed after ${MAX_RETRIES + 1} attempts`, {
          raw,
          error: err instanceof Error ? err.message : String(err),
        });
        throw new Error(`Alert analysis output failed validation: ${err}`);
      }
      // Retry: the LLM gets the previous attempt + error as context
      continue;
    }
  }

  throw new Error("Unreachable");
}
```

**Key rules for retry logic:**
- Retry 2 times. On the third failure, throw and surface to the caller.
- Log the raw output and the validation error on every failure. This is critical for prompt debugging.
- Use `maxIterations: 1` for structured output to prevent the LLM from "helpfully" making tool calls instead of producing the JSON.
- When the schema changes, regenerate the JSON.stringify prompt automatically. A common bug is updating the Zod schema but forgetting to update the prompt.

### Async-First Design

`@slide/agent-core` is fully async-native. All agent operations return `Promise<T>`:

```typescript
// Correct: await the agent run
const result = await runner.run(spec);

// Correct: the Tool.execute() method returns Promise<unknown>
const toolResult = await tool.execute(params);

// Correct: lifecycle hooks are async
class MyHook implements AgentHook {
  async beforeIteration(ctx: AgentHookContext): Promise<void> { /* ... */ }
}
```

**The one common mistake in TypeScript:** Calling `run()` without `await` in an event handler or WebSocket callback, which creates a fire-and-forget promise whose errors are silently swallowed. Always attach `.catch()` or use `try/catch`:

```typescript
// WRONG — unhandled promise rejection
runner.run(spec);

// RIGHT
try {
  const result = await runner.run(spec);
  // handle result
} catch (err) {
  console.error("Agent run failed:", err);
}

// ALSO RIGHT (for fire-and-forget like invoke())
runner.run(spec).catch((err) => {
  console.error("Invoke run failed:", err);
});
```

**Stream vs. await decision:**
| Scenario | Use | Why |
|----------|-----|-----|
| User-facing chat | `hook.wantsStreaming() = true` + `onStream` | Users expect incremental output; full latency is visible |
| AI analysis / alert processing | `hook.wantsStreaming() = false` + `NoopHook` | No user watching; faster completion without streaming overhead |
| Tool-intensive tasks | `hook.wantsStreaming() = true` for first response, then `NoopHook` | Streaming is useful for the first answer; tool execution results can be accumulated silently |
| Structured output validation | `await runner.run()` | Must complete before validation. Stream is wasted since you need the full string. |

### Prompt Engineering Discipline

**System vs. user prompt separation:**
- The **system prompt** defines the agent's persona, capabilities, and constraints. It is always the first message with `role: "system"`.
- The **user prompt** is the current request. It is appended after the system prompt and any conversation history.
- Never concatenate system and user prompts. The message array with distinct roles is the only correct format.

```typescript
// Correct
const messages: Message[] = [
  { role: "system", content: SYSTEM_PROMPT },
  ...history,       // previous turns
  { role: "user", content: userMessage },
];

// Wrong — the model treats this differently from separate roles
const messages: Message[] = [
  { role: "user", content: SYSTEM_PROMPT + "\n\n" + userMessage },
];
```

**Few-shot examples:**
- For **chat**, use inline few-shot in the system prompt (1-3 examples max). Dynamic retrieval is not needed since the prompt length is dwarfed by the conversation.
- For **analysis/invoke**, include 1-2 examples of the expected output format directly in the user message. Together with the Zod schema in the system prompt, this gives the LLM both a formal and an intuitive spec.
- Never hardcode examples that contain sensitive data (real database names, IPs, usernames). Use placeholders like `example-db-01`.

**Token budget:**
- Always set `maxTokens` explicitly. Leaving it unbounded allows the LLM to generate until the context window is full, dramatically increasing cost.
- System prompt budget: aim for <2000 tokens. Longer system prompts dilute attention on the user's actual request.
- Include token count logging in the `LLMProvider.chat()` implementation so per-request cost tracking works from day one.

### Context Window Management

**For Chat (Conversational pattern):**
Context governance is built into `AgentRunner.run()` (microcompact + snip). The developer only needs to:
1. Set `contextWindowTokens` to the model's true limit (e.g., 200K for Claude 4).
2. Keep `maxToolResultChars` reasonable (start at 20000, tune based on actual tool results).
3. The `microcompact` step reduces verbose tool results (read_file, exec) from >10+ turns. If your tools produce results the agent needs to reference later, set `microcompactKeepRecent` higher (editing `MICROCOMPACT_KEEP_RECENT` in runner.ts).

**For Invoke (Autonomous analysis):**
No history management needed — each run is a single-shot or 2-3 turn execution. Keep `maxIterations: 5` to prevent runaway loops.

**For Multi-Agent (future, Phase 109+):**
Each agent instance manages its own context window. Cross-agent communication uses a shared message bus. Agents do not share raw message histories; they share summarised results.

**Summarisation pattern (for conversational overflow):**
If a session reaches the context window limit after snip drops too much history:
```typescript
// Not yet implemented in agent-core; add via checkpointCallback
spec.checkpointCallback = async (payload) => {
  if (payload.phase === "final_response" && /* session too long */) {
    // 1. Request LLM to summarise the conversation so far
    // 2. Replace initial messages with the summary
    // 3. Keep last 2-3 messages for continuity
  }
};
```

## 5. Evaluation Strategy

### Dimensions

Phase 108 is a platform infrastructure refactoring: the five dimensions below verify adapter equivalence — that DirectAdapter, OpenClawAdapter, and the future nanobotAdapter produce identical behavior for the same input. Because the evaluation compares deterministic structures (tool schemas, event sequences, error shapes), all five dimensions use **Code-based measurement**. No LLM judge is required.

| Dimension | Rubric (Pass/Fail) | Measurement Approach | Priority |
|-----------|-------------------|---------------------|----------|
| **Tool Registration Parity** | **PASS:** For every tool in `catalog.ts`, `adapter.listTools()` returns identical name, description, and `parameters` JSON Schema across all adapters. `adapter.capabilities().maxToolCount` matches actual registration count. **FAIL:** A tool is missing from one adapter's list, parameter schemas differ (e.g., `instanceId` vs `instance_id`), or `listTools()` throws for one adapter but returns for another. | **Code** — Deep-equality comparison of `ToolSchema[]` arrays across adapters. Use `JSON.stringify()` on sorted arrays or a structured diff library (e.g., `fast-deep-equal`). Assert count >= `catalog.ts` known tool count. | Critical |
| **Streaming Event Fidelity** | **PASS:** For identical input, both adapters produce the same sequence of `ChatEvent` types (`text_delta`, `tool_start`, `tool_result`, `complete`) with matching ordering and non-empty payloads. No events dropped, duplicated, or reordered. `text_delta` cumulative content converges to the same final text. **FAIL:** Event sequence differs between adapters (e.g., OpenClaw emits `tool_start` before `text_delta` while DirectAdapter emits reverse), any event type is absent, or cumulative text content diverges by more than 10% character count. | **Code** — Capture `ChatEvent[]` arrays from both adapters for same input (using recorded LLM responses for deterministic comparison). Compare event type sequences with array equality. Compare cumulative `text_delta` content with Levenshtein ratio (threshold: >= 0.90 similarity). | Critical |
| **Invoke Reliability** | **PASS:** For identical input and system prompt, `invoke()` on both adapters returns a resolved `InvokeResult` with non-empty `content` string, and the `finalContent` text is semantically equivalent (same key facts, recommendations, and structure). **FAIL:** One adapter rejects the promise while the other succeeds, one returns empty `content`, or outputs contain contradictory facts (e.g., "severity: critical" vs "severity: low" for the same alert input). | **Code** — Compare `InvokeResult.content` across adapters for same recorded input. Assert non-empty on both. Use keyword/entity overlap check for semantic equivalence (extract structured fields from both outputs and compare). Assert no UNKNOWN_ERROR_INVOKE pattern in `content`. | High |
| **Session Continuity** | **PASS:** After a multi-turn chat (3+ turns) followed by an adapter switch, the new adapter produces responses that reference facts from earlier turns (e.g., "the slow query we looked at earlier"). The `chatDatabaseService` history shows all previous turns intact regardless of which adapter produced them. **FAIL:** After adapter switch, the AI has no memory of previous turns, returns contradictory information, or the history view has missing/broken messages. | **Code** — Integration test that: (1) sends 3 turns on adapter A, (2) switches to adapter B via toggle flag, (3) sends a question requiring context from turn 1. Assert the response mentions the earlier context (keyword match). Assert `chatDatabaseService` query returns full history with correct adapter tags. | High |
| **Error Transparency** | **PASS:** When both adapters are given the same intentionally broken input (e.g., tool that always throws, network timeout), both reject with `Error` objects having the same `name` and `message` patterns. The calling platform code catches and displays an actionable message in both cases. **FAIL:** One adapter throws, the other silently succeeds with empty content. Or error messages differ so much that platform-level error handling produces different user-facing messages (e.g., "Server error" vs "Tool execution timed out"). | **Code** — Inject fault conditions (mock tool handler that throws `DatabaseTimeoutError`, mock LLM provider that rejects). Assert both adapters reject with matching error type and message prefix. Assert adapter error includes adapter identifier in error context for audit trail. | High |

**Adapter Parity Test Harness Pattern:**

```typescript
// __tests__/adapter-parity.ts — central test utility for Phase 108 evals
import { describe, it, assert } from "vitest";
import type { IAgentEngine, ToolSchema, ChatEvent, InvokeResult } from "../types";

/**
 * Runs the same input on two adapters and compares results.
 * Uses recorded LLM responses for deterministic comparison
 * (no LLM call during eval — avoids non-determinism).
 */
class AdapterParityTestHarness {
  constructor(
    private adapterA: IAgentEngine,
    private adapterB: IAgentEngine,
  ) {}

  /** Compare listTools() output across adapters. */
  async assertToolParity(): Promise<void> {
    const toolsA = await this.adapterA.listTools();
    const toolsB = await this.adapterB.listTools();

    // Sort by name for deterministic comparison
    const sortByName = (a: ToolSchema, b: ToolSchema) => a.name.localeCompare(b.name);
    toolsA.sort(sortByName);
    toolsB.sort(sortByName);

    assert.strictEqual(toolsA.length, toolsB.length,
      `Tool count mismatch: ${toolsA.length} vs ${toolsB.length}`);

    for (let i = 0; i < toolsA.length; i++) {
      assert.strictEqual(toolsA[i].name, toolsB[i].name,
        `Tool name mismatch at index ${i}`);
      assert.strictEqual(toolsA[i].description, toolsB[i].description,
        `Tool description mismatch for ${toolsA[i].name}`);
      assert.deepStrictEqual(toolsA[i].parameters, toolsB[i].parameters,
        `Tool parameters mismatch for ${toolsA[i].name}`);
    }
  }

  /** Compare streaming event sequences for same input. */
  async assertStreamingParity(
    sessionKey: string,
    input: string,
    mockResponses: { events: ChatEvent[]; result: ChatResult },
  ): Promise<void> {
    // Inject mock LLM responses so both adapters produce the same events
    const eventsA: ChatEvent[] = [];
    const eventsB: ChatEvent[] = [];

    await this.adapterA.chat(sessionKey, input, (e) => eventsA.push(e));
    await this.adapterB.chat(sessionKey, input, (e) => eventsB.push(e));

    // Compare event type sequences
    const typesA = eventsA.map(e => e.type);
    const typesB = eventsB.map(e => e.type);
    assert.deepStrictEqual(typesA, typesB,
      `Streaming event type sequence mismatch`);

    // Compare cumulative text content
    const textA = eventsA.filter(e => e.type === "text_delta")
      .map(e => (e as TextDeltaEvent).delta).join("");
    const textB = eventsB.filter(e => e.type === "text_delta")
      .map(e => (e as TextDeltaEvent).delta).join("");
    assert.isAtMost(
      this.levenshteinRatio(textA, textB),
      0.10, // 10% divergence threshold
      `Cumulative text content divergence exceeds 10%`,
    );
  }

  /** Compare invoke() results. */
  async assertInvokeParity(
    sessionKey: string,
    input: string,
    systemPrompt?: string,
  ): Promise<void> {
    const resultA = await this.adapterA.invoke(sessionKey, input, systemPrompt);
    const resultB = await this.adapterB.invoke(sessionKey, input, systemPrompt);

    assert.ok(resultA.content?.length > 0, `Adapter A invoke returned empty content`);
    assert.ok(resultB.content?.length > 0, `Adapter B invoke returned empty content`);

    // Both should resolve — no silent failures
    // Content comparison is approximate (LLM output varies); check structure
    const hasStructure = (s: string) =>
      s.includes("severity") || s.includes("分析") || s.includes("conclusion");
    assert.ok(hasStructure(resultA.content) || hasStructure(resultB.content),
      `Both invoke results lack expected structure`);
  }

  private levenshteinRatio(a: string, b: string): number {
    const maxLen = Math.max(a.length, b.length);
    if (maxLen === 0) return 0;
    // Simple Levenshtein distance (for strings under 10K chars)
    const dist = this.levenshteinDistance(a, b);
    return dist / maxLen;
  }

  private levenshteinDistance(a: string, b: string): number {
    const m = a.length, n = b.length;
    const dp: number[][] = Array.from({ length: m + 1 }, () => Array(n + 1).fill(0));
    for (let i = 0; i <= m; i++) dp[i][0] = i;
    for (let j = 0; j <= n; j++) dp[0][j] = j;
    for (let i = 1; i <= m; i++) {
      for (let j = 1; j <= n; j++) {
        dp[i][j] = a[i - 1] === b[j - 1]
          ? dp[i - 1][j - 1]
          : 1 + Math.min(dp[i - 1][j], dp[i][j - 1], dp[i - 1][j - 1]);
      }
    }
    return dp[m][n];
  }
}
```

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Adapter initialization failure** | `getAgentEngine()` factory throws during server startup (e.g., DirectAdapter constructor fails after tool registration). | **Fallback**: Log the failure, emit a `server_warning` metric, and return a 503 Service Unavailable from the health endpoint. Do not crash the process. If the previous adapter is still available (e.g., OpenClawAdapter was active before an attempted switch to DirectAdapter), keep the old adapter active and surface the error via `capabilities().status === "degraded"`. |
| **Tool registration completeness check** | After every adapter initialization or switch, the adapter runs an internal `registerTools()` that verifies all tools from `catalog.ts` registered successfully. Missing tool count > 0. | **Block**: Do not accept chat or invoke requests. Report the missing tools via `server.emit("config_error", { missingTools })`. Require manual re-deploy or toggle back to previous adapter via feature flag override. |
| **Stream request stall detection** | `adapter.chat()` is called but no `ChatEvent` (of any type) is emitted within 30 seconds. Or streaming starts (`text_delta` emitted) but then stalls for 60+ seconds without a `complete` or `error` event. | **Escalate**: Abort the streaming session, emit a `chat_timeout` event to the frontend with "Response is taking longer than expected. Please try again or contact support." Log the stalled session key for offline investigation. Do not block future requests — session timeout is per-request, not per-adapter. |
| **Adapter switch consistency check** | Feature flag `ENABLE_AGENT_ADAPTER_CHAT` or `ENABLE_AGENT_ADAPTER_ANALYSIS` is toggled from one adapter to another at runtime. | **Flag**: Before accepting the switch, run a lightweight self-test: call `listTools()` on the new adapter and verify count matches expected tool count from `catalog.ts`. If the check fails, reject the toggle and log the error. Toggle only applies to new sessions — existing in-flight sessions finish on the old adapter. |

**Guardrail latency budget:** Each online guardrail must complete in under 200ms. The stream stall detection timer costs zero CPU — it is a passive timeout. The registration check runs during initialization (not per-request), so it adds no request latency.

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Adapter output divergence rate** | Sample 10% of chat sessions that run through both adapters concurrently (dual-run during Plan 3 validation period). Compare event sequences and final output. Target: 100% identical event type sequences, >90% Levenshtein similarity on final text. | If divergence rate exceeds 5% of sampled sessions: escalate to platform engineering. Determine whether the divergence is caused by non-deterministic LLM output (acceptable) or adapter-specific event translation bugs (requires fix). Auto-pause dual-run if divergence > 20% to avoid unnecessary cost. |
| **Silent error rate per adapter** | Sample 100% of `invoke()` calls that return empty content or non-standard error paths. Track per-adapter: count of resolved vs rejected promises, count of empty-content results. Sampled every hour. | If an adapter's empty-content rate exceeds 2% of invoke calls for 3 consecutive hours: auto-toggle to the other adapter for analysis tasks (`ENABLE_AGENT_ADAPTER_ANALYSIS`), generate a P2 ticket, and flag the degraded adapter for investigation. |
| **Session continuity violation rate** | Sample 1% of multi-turn sessions (defined as 5+ consecutive messages) by querying `chatDatabaseService` for sessions with mixed adapter tags. Verify every message has a valid adapter tag and that no messages are missing in the sequence. Sampled daily. | If >0.5% of sampled multi-turn sessions show gaps or missing adapter tags: generate a P3 ticket for review. The internal audit trail (regulatory requirement per Section 1b) must be complete. |
| **Tool registration drift** | Weekly automated check: run `listTools()` against both adapters and compare with the canonical tool list from `catalog.ts`. Log any mismatch. | If any mismatch is detected: generate a P2 ticket. The mismatch indicates a deployment gap (a new tool was added to catalog.ts but not registered on one adapter). Fix before the next production deploy. |

## 7. Production Monitoring

**Tracing Tool:** Structured JSON logging + OpenTelemetry (optional, for future Phoenix deployment)

Phase 108 is infrastructure — the primary monitoring signal is **adapter behavior parity**, not LLM response quality. Since the project is TypeScript throughout and no Python tracing stack is currently deployed, start with structured JSON logging that can be ingested by any log aggregator (ELK, Grafana Loki, or Datadog). Each adapter operation emits structured log lines with consistent fields. If tracing depth is needed later, add OpenTelemetry JS SDK (`@opentelemetry/sdk-trace-node` + `@opentelemetry/exporter-otlp-http`) and point it at a self-hosted Arize Phoenix instance (`px.launch_app()` on port 6006).

**Structured log format for adapter operations:**

```typescript
// Every adapter.chat(), adapter.invoke(), and adapter.listTools() call logs:
{
  timestamp: "2026-05-25T10:00:00.000Z",
  level: "info",
  event: "adapter.chat" | "adapter.invoke" | "adapter.listTools",
  adapter: "direct" | "openclaw" | "nanobot",
  sessionKey: "sess_abc123",
  durationMs: 1234,
  eventCount: 5,              // only for chat: ChatEvent count
  tokenUsage: { input: 1500, output: 300 },  // if available
  error: null | { name: "TimeoutError", message: "..." }
}
```

**Key Metrics to Track:**

1. **Adapter error rate** — Count of rejected promises / total calls per adapter. Track separately for `chat()` and `invoke()`. Alert if > 1% for any 5-minute window.
2. **Empty invoke result rate** — Count of `invoke()` calls returning empty `content` or empty `InvokeResult`. Alert if > 0.5% in any hour (silent analysis failures are invisible to users).
3. **Adapter switch frequency** — Count of feature flag toggles per day. A spike may indicate users are workarounding an adapter-specific issue.
4. **Streaming stall rate** — Count of `chat()` calls that timeout (no events within 30s) or stall (incomplete stream). Alert if > 2% of chat sessions.

**Alert Thresholds:**

| Alert | Condition | Severity | Response |
|-------|-----------|----------|----------|
| Adapter initialization failure | `getAgentEngine()` throws on startup | P0 (critical) | Engineer pages immediately. Service is degraded or down for AI features. |
| Adapter error rate > 2% | Rolling 5-min window, any adapter | P1 (high) | Engineer pages within 15 minutes. Auto-toggle analysis to the other adapter if available. |
| Empty invoke rate > 1% | Rolling 1-hour window | P2 (medium) | Next-business-day investigation. May indicate silent adapter regression. |
| Tool registration drift | Weekly check detects mismatch | P2 (medium) | Schedule fix before next deploy cycle. |
| Session continuity violation | Daily check detects gaps | P3 (low) | Logged for post-incident review. |

**Smart Sampling Strategy:**

Since Phase 108 is infrastructure (not content generation), "human review" means inspecting structured logs for adapter behavior anomalies, not reading LLM output for quality. Sample using signal-based filters:

1. **Always sample**: Any chat session where `adapter.chat()` produced 0 events, or `invoke()` returned empty content. These are potential silent failure cases.
2. **Always sample**: Any session that crossed an adapter boundary (feature flag toggle mid-conversation). These are the highest-risk operations for session continuity.
3. **Rate-limit sample (1 in 100)**: All chat sessions on the default adapter (DirectAdapter) during steady state. Provides baseline for drift detection.
4. **Rate-limit sample (1 in 10)**: All chat sessions on non-default adapters (OpenClawAdapter) during the dual-run validation period (Plan 3). Provides coverage for adapter equivalence verification.
5. **Adversarial sample trigger**: If `ENABLE_AGENT_ADAPTER_CHAT` is toggled more than 3 times in an hour, sample 100% of sessions for the next 30 minutes. Frequent toggling suggests a user-side workaround pattern.

import { NodeTracerProvider } from "@opentelemetry/sdk-trace-node";
import { OTLPTraceExporter } from "@opentelemetry/exporter-otlp-http";
import { BatchSpanProcessor } from "@opentelemetry/sdk-trace-base";

export function initTelemetry(endpoint = "http://localhost:6006/v1/traces") {

  const tracer = trace.getTracer("slide-adapter");

  return { tracer, provider, exporter };
}
