<task type="auto">
  <name>Task 2: Enhance EXPLAIN normalizer for edge-case formats</name>
  <files>
    <read_first>frontend/src/openclaw/ui/views/sql-console.ts</read_first>
    <change>frontend/src/openclaw/ui/views/sql-console.ts</change>
  </files>
  <action>
    The current normalizer handles standard MySQL 8.0 `query_block` and MariaDB `query_plan` formats.
    Add handling for three additional variants:

    **2a. MySQL 5.7 EXPLAIN FORMAT=JSON (records without nested_loop, uses `rows` directly):**
    In `_normalizeMySQLNode()` (line ~707), after the existing accessType/operation logic, add:
    - Check if `node.rows` exists (numeric, not undefined) and use it as the rows count.
    - Check if `node.cost_info?.query_cost` is missing but `node.query_cost` exists (flat format).
    - Some MySQL 5.7 outputs put `possible_keys`, `key`, `key_len`, `ref`, `rows`, `Extra` at the same level as `table`.

    **2b. MariaDB `query_plan` format with flat table structure:**
    In `_normalizeMySQLPlan()` (line ~690), the existing `query_plan` handler already handles `qp.operation`, `qp.estimated_rows`, `qp.estimated_total_cost`, and `qp.index_used`. Add:
    - `qp.type` as fallback operation name if `qp.operation` is missing.
    - `qp.rows` as fallback for `qp.estimated_rows`.
    - `qp.cost` as fallback for `qp.estimated_total_cost`.

    **2c. PostgreSQL planner-extended format (`SubplanScan`, `ModifyTable`):**
    In `_normalizePGNode()` (line ~753), add handling for:
    - `Subquery Scan` and `Subplan` node types (have `Subplan Name`).
    - `ModifyTable` type (has `Operation` field inside the node).
    - Handle `Subplan Name` in details: `if (node['Subplan Name']) details['Subplan'] = node['Subplan Name'];`.
    - Handle `Operation` for ModifyTable: `if (node['Operation']) details['Operation'] = node['Operation'];`.

    **2d. Generic fallback:** If the plan input is neither recognized format, try to detect:
    - If plan has a `query` field (Aurora MySQL variant), recurse into it.
    - If plan has `execution_plan` or `executionPlan`, recurse into that.
    - If plan is an array, normalize the first element and merge additional array items as children.

    Per D-06: Handle edge cases where MySQL/PostgreSQL EXPLAIN JSON variants don't match the existing normalizer patterns. The normalizer should never crash — unknown formats should return `{operation: 'unknown', rows: 0, cost: 0, children: [], details: {}}`.
  </action>

  </verify>

    - MariaDB: qp.type and qp.rows are recognized as fallbacks
    - PostgreSQL: SubplanScan, ModifyTable, Subplan Name handled
    - Generic fallback: query, execution_plan, array formats handled
