<objective>
Add database/schema selector support to SQL execution and approval auto-execution.

**Purpose:** SQL Console currently only has instance selector; approval auto-exec DDL goes to default connection database (e.g. `mysql` system db instead of `db_ops_ai`). This plan adds a `database_name` parameter through the entire execution chain.

**Output:**
- Migration `003_add_target_database.sql` — adds `target_database VARCHAR(128)` to `approval_requests`
- `sql-executor.ts` — `executeSql()` accepts optional `database` param, executes `USE \`{db}\`` before SQL
- `approval-service.ts` — `submitForApproval()` stores `target_database`, `reviewRequest()` passes it to `executeSql`
- `server.ts` — execute route and approval submit route accept and forward `database` / `database_name`
</objective>

  <action>
    Modify `executeSql()` to accept an optional `database: string` in the `context` parameter.

    1. Add `database?: string` to the `context` parameter type shape (or extend the destructuring).
    2. After getting the connection (`conn`) but BEFORE executing the SQL:
       - If `context.database` is provided and `conn.db_type === 'mysql'`:
         Execute `await conn.pool.query('USE \`' + context.database.replace(/`/g, '``') + '\`')` to switch databases.
         (Backtick-escape the database name to prevent SQL injection through the database name param.)
       - If `context.database` is provided and `conn.db_type === 'postgresql'`:
         Execute `await conn.pgClient.query('SET search_path TO ' + conn.pgClient.escapeIdentifier(context.database))`.
       - For other db_types: log a warning and proceed (database switching not supported for that type).
    3. Do NOT modify the return type or the existing SQL execution flow. Only add the database-switch step.

    No test changes needed — the existing tests should still pass with the backwards-compatible optional param.
  </action>
  <verify>

  </verify>

    - MySQL path executes `USE database_name` with backtick-escaped name before query
    - PostgreSQL path executes `SET search_path TO database_name` before query
    - The change is backwards compatible (existing calls without database param work as before)
  </acceptance_criteria>
