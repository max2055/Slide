    **Root cause:** schema collect/detect routes use `requirePermission()` and `requireInstanceAccess()` middleware. When these return 403 Fastify error responses, the frontend's `res.json()` receives a non-JSON error page or the response is not parsed correctly, leading to generic "Bad Request" display.

    Also, the collect route (line 2269-2283) and detect/changes route (line 2331-2345) both have these permissions. The collect route returns `result.error` from schemaService, while the detect route returns `data.error` on failure.

    **Server fix (server.ts):** Verify that the schema collect/detect route handlers properly catch all error types:

    1. Around the collect route handler (line ~2271):
       - Add a try/catch for the `requirePermission`/`requireInstanceAccess` failure path. If the user doesn't have `schema:manage` permission, the Fastify preHandler returns a 403. The frontend fetch response will have `ok: false` and status 403. The frontend should handle non-OK responses.
       - Currently the collect route returns `reply.code(400).send({ success: false, error: result.error })` for schemaService errors. Add handling for preHandler failures.

       Actually, the preHandler failures (403) are handled by Fastify's built-in error handler, which returns `{"statusCode":403,"error":"Forbidden","message":"...}"`. The frontend's `res.json()` will work but `data.success` will be undefined. So the frontend needs to handle non-OK responses.

    2. In the detect route (line ~2331):
       - Same issue: preHandler 403 error responses are JSON format but frontend parsing doesn't handle them.

    **Frontend fix (schema-management.ts):**

    1. Update `handleCollect()` and `handleDetect()` methods to handle non-OK HTTP responses:
       - Instead of always calling `res.json()`, check `res.ok` first:
         ```typescript
         if (!res.ok) {
           const err = await res.json().catch(() => ({ error: `HTTP ${res.status}` }));
           this.error = err.error || `请求失败 (${res.status})`;
           return;
         }
         ```
       - Apply this pattern to both methods.

    2. Update `handleDetect()` (line ~543-561) and `handleCollect()` (line ~521-541).

    Per D-04: HTTP error responses from permission middleware should show readable messages, not generic "Bad Request".
  </action>
