<objective>
Install dmdb driver, migrate all Dameng connection paths from oracledb to dmdb, create separate Dameng-only active sessions and capacity methods (splitting from Oracle shared paths), add Dameng to metric registry and agent tools

Purpose: Phase 95 requires full Dameng DM8 database management. The codebase currently has 'dameng' in type unions and skeleton methods that use oracledb (via Dameng's Oracle compatibility mode). Per decision D-05, this replaces oracledb with the official dmdb npm package for native Dameng protocol support. Per the architectural decision, this plan creates separate `getDamengActiveSessions()` and `getDamengCapacity()` methods using `conn.dmConnection`, splitting from the previously shared Oracle+Dameng code paths. Also per D-02 and D-04, adds Dameng to the metric registry and agent tool type unions, plus adds schema object browsing support so the SQL console schema browser works for Dameng instances.

Output:
- dmdb dependency installed and imported
- All Dameng connection paths in database-service.ts migrated from oracledb to dmdb
- New `getDamengActiveSessions()` method using `conn.dmConnection` querying `V$SESSIONS`
- New `getDamengCapacity()` method using `conn.dmConnection` querying `DBA_DATA_FILES` / `DBA_SEGMENTS`
- `getActiveSessions()` dispatch split: `'dameng'` -> `getDamengActiveSessions()`, `'oracle'` -> `getOracleActiveSessions()`
- `getCapacityInfo()` dispatch split: `'dameng'` -> `getDamengCapacity()`, `'oracle'` -> `getOracleCapacity()`
- All Oracle (oracledb) code paths completely untouched
- testConnection migrated in instance-database-service.ts
- getSchemaObjects supports Dameng via ALL_TAB_COLUMNS
- metric-registry has 'dameng' in db_types for common metrics
- add_database.ts agent tool includes 'dameng' in db_type union
- test_connection.ts agent tool includes 'dameng' in db_type enum + dmConnection fast-path

### Part C: Create getDamengActiveSessions() -- new Dameng-only method

Create a new private method `getDamengActiveSessions()` near the existing `getOracleActiveSessions()` (after line ~2284). Do NOT modify `getOracleActiveSessions()`.

```
private async getDamengActiveSessions(conn: DatabaseConnection): Promise<any[]> {
  if (!conn.dmConnection) return [];

  const result = await conn.dmConnection.execute(`
    SELECT
      s.SESS_ID as id,
      s.USERNAME as user,
      s.CLNT_IP as host,
      s.APPNAME as program,
      s.STATE as state,
      s.SQL_TEXT as query,
      s.LAST_SEND_TIME as time_seconds
    FROM V$SESSIONS s
    WHERE s.STATE = 'ACTIVE'
      AND s.SESS_ID != (
        SELECT SESS_ID FROM (
          SELECT SESS_ID, ROWNUM AS rn FROM V$SESSIONS WHERE USERNAME = 'SYSDBA' AND APPNAME = 'MONITOR'
        ) WHERE rn = 1
      )
    ORDER BY s.LAST_SEND_TIME DESC NULLS LAST
    FETCH FIRST 50 ROWS ONLY
  `);

  return (result.rows || []).map((row: any) => ({
    id: row[0],
    serial_num: null,
    user: row[1],
    host: row[2],
    program: row[3],
    state: row[4],
    sql_id: null,
    query: row[5],
    time_seconds: row[6],
  }));
}
```

Key differences from `getOracleActiveSessions()`:
- Uses `conn.dmConnection` not `conn.oracleConnection`
- Queries `V$SESSIONS` (Dameng) not `v$session` (Oracle)
- Maps `SESS_ID` instead of `SID`, `CLNT_IP` instead of `MACHINE`, `APPNAME` instead of `PROGRAM`
- SQL_TEXT comes directly from V$SESSIONS (no JOIN needed -- Dameng stores SQL text inline)
- Excludes self via a ROWNUM-based approach instead of Oracle's `v$mystat` (which does not exist in Dameng)
- `serial_num` and `sql_id` set to `null` since Dameng does not have direct equivalents

```
private async getDamengCapacity(conn: DatabaseConnection): Promise<any> {
  if (!conn.dmConnection) return null;

  // 获取表空间信息 -- DBA_DATA_FILES is compatible between Oracle and Dameng
  const tsResult = await conn.dmConnection.execute(`
    SELECT
      tablespace_name as name,
      ROUND(SUM(bytes) / 1024 / 1024 / 1024, 2) as size_gb,
      ROUND(SUM(DECODE(autoextensible, 'YES', maxbytes, bytes)) / 1024 / 1024 / 1024, 2) as max_size_gb,
      ROUND((SUM(bytes) / SUM(DECODE(autoextensible, 'YES', maxbytes, bytes))) * 100, 2) as usage_percent
    FROM dba_data_files
    GROUP BY tablespace_name
    ORDER BY size_gb DESC
  `);

  // 获取最大的段（表/索引）-- DBA_SEGMENTS is compatible between Oracle and Dameng
  const segResult = await conn.dmConnection.execute(`
    SELECT
      owner || '.' || segment_name as name,
      segment_type as type,
      ROUND(SUM(bytes) / 1024 / 1024 / 1024, 2) as size_gb
    FROM dba_segments
    WHERE segment_type IN ('TABLE', 'INDEX')
    GROUP BY owner, segment_name, segment_type
    ORDER BY size_gb DESC
    FETCH FIRST 20 ROWS ONLY
  `);

  const tablespaces = (tsResult.rows || []).map((row: any) => ({
    name: row[0],
    size_gb: Number(row[1]),
    max_size_gb: Number(row[2]),
    usage_percent: Number(row[3]),
  }));

  const totalSize = tablespaces.reduce((sum: number, ts: any) => sum + (ts.size_gb || 0), 0);

  return {
    total_size_gb: Math.round(totalSize * 100) / 100,
    tablespaces,
    top_segments: (segResult.rows || []).map((row: any) => ({
      name: row[0],
      type: row[1],
      size_gb: Number(row[2]),
    })),
  };
}
```

Key differences from `getOracleCapacity()`:
- Uses `conn.dmConnection` not `conn.oracleConnection`
- Query structure identical (DBA_DATA_FILES and DBA_SEGMENTS are compatible views)
- Return format identical (consumed by same frontend components)
