## UI Changes Summary

Phase 95 introduces **zero new UI components**. All Dameng functionality is achieved through:

1. **CodeMirror dialect addition** (`sql-console.ts`): Add `DamengDialect` via `SQLDialect.define()`, switch to it when instance db_type is 'dameng'. No visual changes to UI.

2. **Instance detail tab content** (`instance-detail.ts`): The existing 6 tabs (Overview, Monitor, TopSQL, Slow Query, Capacity, Sessions) remain unchanged. Data source switches to Dameng backend endpoints. The `MetricsData` type already includes `dm_*` fields. No template changes needed.

3. **Add-instance form** (`instances-db.ts`): Already has db_type='dameng' in dropdown. Port 5236 is already set as default for dameng. No form changes needed.

4. **Instance list table** (`instances-db.ts`): Already renders db_type column. 'dameng' displays as-is in the type badge. No changes needed.

## Interaction Contracts

| Interaction | Behavior | Notes |
|-------------|----------|-------|
| SQL dialect switching | When user selects a Dameng instance in SQL console, DamengDialect activates via `@codemirror/lang-sql` | Same pattern as existing MySQL/Oracle switching |
| CodeMirror autocompletion | Custom completion source provides `V$DM_*` views, `DM_SQL_*` functions, and Dameng system views | Added via `autocompletion({ override: [sqlCompletionSource] })` |
| Metric polling | Metrics endpoint returns `dm_*` fields for Dameng instances; displayed in same metric charts as MySQL/Oracle | Backend change only; frontend chart components are db_type-agnostic |
| AI agent tools | db_type='dameng' added to tool type unions; tools operate on Dameng instances transparently | Backend change only; frontend chat component unchanged |
