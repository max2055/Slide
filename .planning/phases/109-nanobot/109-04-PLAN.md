Remove Gateway protocol dependency from frontend and adapt to DirectAdapter native WS protocol.

**Purpose:** The frontend currently communicates with the backend via OpenClaw Gateway protocol (GatewayBrowserClient in `gateway.ts`, protocol schemas in `protocol/`). **Scope boundary (per research):** Only the Chat path. Non-chat openclaw/ files (device auth, i18n, styles, app-render, navigation, sidebar, chat rendering components) are kept untouched. Full openclaw/ cleanup is Phase 112.

**Output:** New frontend WS client, adapted app-gateway.ts, removed chat-only protocol schema files
</objective>

- GatewayBrowserClient class — manages WS connection, auth, heartbeat, request/response frame protocol
- GatewayEventFrame { type: 'event', event: string, payload?, seq? }
- GatewayResponseFrame { type: 'res', id: string, ok: boolean, payload?, error? }

- 650 lines — the main WS event handler
- Imports from gateway.ts (GatewayBrowserClient, GatewayEventFrame, GatewayResponseFrame)
- Imports controllers: chat, sessions, health, agents, exec-approval
- Handles Gateway protocol events (tick, shutdown, presence, etc.)
- Routes chat events to chat controller

DirectAdapter WS protocol (from Part 3):
- Client sends: {"type":"chat.send","sessionKey":"...","message":"..."}
- Client sends: {"type":"chat.history","sessionKey":"..."}
- Server sends: ChatEvent JSON (type: text_delta, tool_start, tool_result, tool_error, complete, error)
- Port: AGENT_WS_PORT (default 28789)
- No auth handshake, no heartbeat, no request/response frames — just chat.send + ChatEvent stream

ChatEvent types (from adapter/types.ts):
- TextDeltaEvent { type: 'text_delta', delta: string }
- ToolStartEvent { type: 'tool_start', toolName: string, args: Record<string, unknown> }
- ToolResultEvent { type: 'tool_result', toolName: string, result: unknown }
- ToolErrorEvent { type: 'tool_error', toolName: string, error: string }
- CompleteEvent { type: 'complete', finalContent?: string }
- ErrorEvent { type: 'error', error: string }

    1. **direct-gateway.ts** (NEW): Create `DirectGatewayClient` class:
       ```typescript
       export class DirectGatewayClient {
         private ws: WebSocket | null = null;
         private url: string;
         private onEvent: (event: ChatEventPayload) => void;
         private onStateChange: (state: 'connecting' | 'connected' | 'disconnected') => void;
         private reconnectTimer: ReturnType<typeof setTimeout> | null = null;
         private reconnectAttempts = 0;
         private maxReconnectAttempts = 10;

         constructor(opts: {
           url?: string;  // default: `ws://${location.hostname}:28789`
           onEvent: (event: ChatEventPayload) => void;
           onStateChange: (state: 'connecting' | 'connected' | 'disconnected') => void;
         });

         connect(): void — create WebSocket, register onmessage/onclose/onerror handlers
           - onmessage: parse JSON, dispatch to onEvent
           - onclose: auto-reconnect with exponential backoff (1s, 2s, 4s... up to 30s)
         disconnect(): void — close WebSocket, clear reconnect timer
         sendChat(sessionKey: string, message: string): void
           - Send: JSON.stringify({ type: 'chat.send', sessionKey, message })
         requestHistory(sessionKey: string): void
           - Send: JSON.stringify({ type: 'chat.history', sessionKey })
         isConnected(): boolean
       }
       ```
       The `onEvent` callback receives a ChatEventPayload — this is the same union type used by the frontend chat controller. Import from the existing frontend's chat controller types.

       Key design: This is a THIN wrapper — no auth handshake, no heartbeat, no request/response frame IDs. Just the DirectAdapter native protocol.

    2. **gateway.ts** (MODIFY): Add a deprecation re-export at the top that explains migration:

    3. **app-gateway.ts** (MODIFY): Adapt to support DirectAdapter native protocol:

       a. Add import for DirectGatewayClient:

       b. Add a `FEATURE_GATEWAY_ENABLED` flag check or detection logic:
          At startup, try connecting to DirectAdapter WS first. If it connects, use DirectGatewayClient. If not, fall back to existing GatewayBrowserClient. Detection:

          - Send `{ type: 'chat.history', sessionKey: 'detect' }`
          - If response matches ChatEvent format (has type field matching ChatEvent union) -> DirectAdapter mode
          - Otherwise -> Gateway mode

          Simpler approach: check for `window.__SLIDE_USE_DIRECT_ADAPTER` or env config endpoint. Or just try DirectAdapter first and fall back.

          - `tool_error` -> calls handleAgentEvent

          IMPORTANT: This mapping is the key adaptation point. Study existing app-gateway.ts event handling chain to reuse the same dispatch functions. Do NOT rewrite the chat rendering.
