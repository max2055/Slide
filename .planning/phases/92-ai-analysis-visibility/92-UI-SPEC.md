## New / Refactored Specifications

### 1. Analysis Status Badge (in Alert List Table)

**Location:** Column "分析状态" in `alerts.ts` alert table. Current inline rendering (lines 979-992) replaced with a reusable rendering method.

**Specification:**
- Pill shape: inline-flex, `--space-xs` (4px) vertical padding, `--space-md` (12px) horizontal padding
- Border radius: `--radius-full` (9999px)
- Font: `--text-xs` (11px), weight 600
- States:

| State | CSS Classes | Visual |
|-------|------------|--------|
| `completed` | `analysis-badge analysis-badge--completed` | Green bg/text pill: "已分析" |
| `running` | `analysis-badge analysis-badge--running` | Violet bg/text pill: "分析中" |
| `failed` | `analysis-badge analysis-badge--failed` | Violet-danger bg/text pill: "分析失败" |
| No analysis | (no badge) | Em dash "—" in muted color |

**Interaction:**
- `completed` badge is **clickable** -> opens analysis result modal (see Section 2)
- `running` badge is non-interactive, shows spinner optional (but pill is sufficient)
- `failed` badge is **clickable** -> opens error detail modal showing failure reason
- No-analysis state is non-interactive

**Implementation note:** Reuse `status-badge-sm` pattern from alerts.ts (lines 617-636). The boolean `analyzedAlertIds.has(alert.id)` check on line 956 already exists; extend to also map `status` values from the API response rather than just existence.

### 2. Unified `ai-analysis-result` Lit Web Component (Refactored)

**Location:** `frontend/src/openclaw/ui/views/ai-analysis-result.ts`

**Current state:** Not data-driven — manages its own API calls and polling. Must be refactored per D-07.

**New interface (Lit properties):**

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `result` | `string \| null` | `null` | Markdown content from `ai_analysis.result` column |
| `analysisType` | `string` | `"alert_rca"` | One of: `alert_rca`, `fault_diagnosis`, `topsql_analysis` |
| `triggerType` | `string` | `"manual"` | `auto` or `manual` — controls source tag |
| `loading` | `boolean` | `false` | Show loading skeleton when true |
| `status` | `string` | `"completed"` | `pending`, `running`, `completed`, `failed` |
| `errorMessage` | `string \| null` | `null` | Error message when status is `failed` |
| `title` | `string` | `"AI 分析结果"` | Card title override for different contexts |

**Remove:** `analysisId`, `pollTimer`, `startAnalysis()`, `_startPolling()`, `_stopPolling()` — these move to parent components (alerts.ts / instance-detail.ts).

**Render states:**

| State | Visual |
|-------|--------|
| `loading=true` | Spinner + "AI 分析中，请稍候..." centered in card body |
| `status="running"` | Animated pulse indicator + "分析中..." text (reuse spinner) |
| `status="failed"` | Red error card with error icon + `errorMessage` |
| `status="completed"` + `result=null` | Muted text: "分析完成，但暂无结果数据" |
| `status="completed"` + `result` string | Rendered Markdown via `marked.parse()` |
| `status="completed"` + `result` object | Rendered via existing JSON formatter (backward compat) |

**Source tag display:**
- Position: Top-right corner of card header, next to close button
- Style: Small pill/tag using trigger source colors (see Color section)
- Icon (optional): `brain` icon for auto, `zap` icon for manual

**Markdown rendering rules:**
- Use `marked.parse(result, { async: false })` to convert Markdown to HTML
- Wrap output in `<div class="result-content">` with existing styles
- **XSS prevention:** Sanitize `marked` output. The `marked` library by default does NOT sanitize HTML. Apply DOMPurify or at minimum strip `<script>` tags. Follow the `esc()` pattern from `instance-detail.ts` (line 1382) as a baseline.
- Code blocks: Render with `--mono` font, `--bg-elevated` background, horizontal scroll for wide content
- Tables in Markdown: Ensure `marked` generates `<table>` with proper class. Add base CSS: `table { border-collapse: collapse; width: 100%; } th, td { border: 1px solid var(--border); padding: var(--space-sm); }`
- Heading levels: `##` -> card section headers with `--text-md` weight 600, `###` -> subsection headers with `--text-base` weight 600

## Edge Cases

| Edge Case | Behavior |
|-----------|----------|
| **Long Markdown content** | Modal has `max-height: 85vh` with `overflow-y: auto`. Code blocks get `overflow-x: auto` with horizontal scroll. |
| **Result column stores JSON-wrapped string** | The `result` from API may be `JSON.stringify(markdownString)` wrapping. The component must handle: try `marked.parse(result)`, if result looks like JSON-escaped string, try `JSON.parse(result)` first. |
| **Missing ai_analysis record** | API returns 404 -> show "无分析结果" muted text, not crash |
| **Null/undefined in numeric fields** | Use `(value ?? 0).toFixed(n)` pattern per folded todo (Research Pitfall 5) |
| **Long diagnosis summary text** | Truncate at 80 chars with ellipsis. Full content available in modal. |
| **Empty whitelist** | Means "all instances". Show help text: "全部实例参与分析" |
| **XSS from Agent-generated Markdown** | Sanitize `marked` output. Strip `<script>`, `<iframe>`, `on*` event handlers. Use DOMPurify or minimal regex sanitization. |
| **Polling timeout (5 min)** | Show "分析超时，请稍后重试" error. Status: `failed`. |
| **Network error during result fetch** | Show "加载分析结果失败，请刷新重试" with retry button. |
| **Zero recent diagnoses** | Diagnosis history section shows empty state: icon + "暂无诊断记录" |
| **Very long instance whitelist** | Tag list wraps to next line. Each tag is a truncatable pill. |
| **Invalid cron expression** | Client-side validation: alert "Cron 表达式格式无效" before submit. Use simple regex or existing cron parser validation on backend. |
| **Token key mismatch** | Fix `ai-analysis-result.ts` to use `localStorage.getItem("token")` not `"slide_token"` (Pitfall 2 from Research). |
| **Silent _loadAnalyzedStatuses failure** | Add `console.warn()` in catch block per D-12. |

## Accessibility Notes

- Analysis status badges: Use `<span>` with `aria-label` (e.g., `aria-label="AI 分析已完成"`) — no color-dependent information
- Modal: Trap focus inside modal while open. Close on ESC key.
- Toggle switch: Use native `<input type="checkbox">` styled as toggle (existing `cfg-toggle` pattern)
- Color contrast: All badge text/background combinations must pass WCAG AA. The current custom colors are fine (violet on dark bg, green on dark bg).
- Click targets: Badges and summary rows should have minimum 32px height for touch targets on mobile.
