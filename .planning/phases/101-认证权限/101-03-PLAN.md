**Step A: Add private attemptTokenRefresh() method:**

This is the core refresh logic:
```typescript
private async attemptTokenRefresh(): Promise<string | null> {
  const refreshToken = this.getRefreshToken();
  if (!refreshToken) return null;

  try {
    const path = this.baseURL.startsWith('http') 
      ? this.baseURL + '/auth/refresh'
      : '/api/auth/refresh';
    
    const response = await fetch(path, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ refreshToken }),
    });

    if (!response.ok) {
      // Refresh failed — clear everything, user must re-login
      this.setToken(null);
      this.setRefreshToken(null);
      return null;
    }

    const data = await response.json();
    this.setToken(data.token);
    this.setRefreshToken(data.refreshToken);
    return data.token;
  } catch {
    // Network error — don't clear tokens, might succeed later
    return null;
  }
}
```

Note: The refresh call uses plain `fetch()` — NOT the ApiClient's own methods — to avoid recursion (the refresh endpoint itself doesn't need auth per the plan's design; D-04 says refresh endpoint validates the refresh token, not the access token).

**Step B: Refactor get/post/put/delete to use a shared fetchWithAuth method.**

Add private method:
```typescript
private async fetchWithAuth<T>(url: string, options: RequestInit): Promise<T> {
  const executeFetch = (token: string | null) => {
    const headers = new Headers(options.headers || {});
    if (!headers.has('Content-Type')) {
      headers.set('Content-Type', 'application/json');
    }
    if (token) {
      headers.set('Authorization', `Bearer ${token}`);
    }
    return fetch(url, { ...options, headers });
  };

  let response = await executeFetch(this.getToken());

  if (response.status === 401) {
    // Deduplicate concurrent refresh attempts using promise chain
    if (!this.refreshPromise) {
      this.refreshPromise = this.attemptTokenRefresh().finally(() => {
        this.refreshPromise = null;
      });
    }
    const newToken = await this.refreshPromise;
    if (newToken) {
      response = await executeFetch(newToken);
    }
  }

  return this.parseResponse<T>(response);
}
```

**Step C: Refactor all four HTTP methods** to delegate to fetchWithAuth.

Each method becomes:
```typescript
async get<T>(endpoint: string, config?: RequestConfig): Promise<T> {
  const url = this.buildURL(endpoint, config?.params);
  return this.fetchWithAuth<T>(url, { method: 'GET', ...config });
}

async post<T>(endpoint: string, data?: unknown, config?: RequestConfig): Promise<T> {
  const url = this.buildURL(endpoint);
  return this.fetchWithAuth<T>(url, {
    method: 'POST',
    body: data ? JSON.stringify(data) : undefined,
    ...config,
  });
}

async put<T>(endpoint: string, data?: unknown, config?: RequestConfig): Promise<T> {
  const url = this.buildURL(endpoint);
  return this.fetchWithAuth<T>(url, {
    method: 'PUT',
    body: data ? JSON.stringify(data) : undefined,
    ...config,
  });
}

async delete<T>(endpoint: string, config?: RequestConfig): Promise<T> {
  const url = this.buildURL(endpoint, config?.params);
  return this.fetchWithAuth<T>(url, { method: 'DELETE', ...config });
}
```

CRITICAL: The refactored methods must NOT include `headers` in the top-level object — the fetchWithAuth method handles Authorization and Content-Type. Passing config.headers as part of spread (...config) is acceptable; headers that come via RequestConfig will be merged by the Headers constructor in fetchWithAuth.

DO NOT change the public API signatures, constructor, or any other methods. Do NOT modify parseResponse or buildURL. The existing behaviors must be preserved — the only difference is automatic 401 retry.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser localStorage | Access token + refresh token stored client-side; XSS could exfiltrate |
| Browser -> /api/auth/refresh | Refresh token sent in plaintext over HTTPS (in dev: HTTP localhost) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-101-10 | Information Disclosure | localStorage tokens | accept | Internal tool with no PII; XSS risk is mitigated by Content Security Policy. Accept per internal tool risk profile. |
| T-101-11 | Tampering | 401 interceptor recursion | mitigate | attemptTokenRefresh() uses plain fetch() not this.fetchWithAuth() — no recursion risk. |
| T-101-12 | Denial of Service | Race condition on 401 retry | mitigate | refreshPromise pattern ensures only one /api/auth/refresh call at a time. All concurrent 401s queue behind the same promise. |
| T-101-13 | Tampering | Refresh token via non-HTTPS | accept | Development only (localhost). Production assumes HTTPS through proxy. |

</threat_model>
