**Part A: app-settings.ts**

Add a `hasSlidePermission` function after the existing `hasOperatorReadAccess()` function (around line 508):
```typescript
export function hasSlidePermission(
  permissions: Set<string> | null | undefined,
  required: string | undefined,
): boolean {
  if (!required) return true; // No requirement = always visible
  if (!permissions) return false;
  // Wildcard matching
  if (permissions.has('*')) return true;
  if (permissions.has(required)) return true;
  // Resource-level wildcard: instance:view matches permission set containing instance:*
  const colonIdx = required.indexOf(':');
  if (colonIdx !== -1) {
    const resourcePrefix = required.substring(0, colonIdx) + ':*';
    if (permissions.has(resourcePrefix)) return true;
  }
  return false;
}
```

Export it. Do NOT modify existing `hasOperatorReadAccess()`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser localStorage | Cached permissions stored client-side; XSS could modify |
| Browser -> GET /api/auth/permissions | Authenticated request for current user's permission set |
| Sidebar rendering | Permission check determines tab visibility |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-101-14 | Elevation of Privilege | localStorage permission modification | accept | Permissions in localStorage are only used for UI hints (show/hide tabs). Actual authorization is enforced server-side by requirePermission() and requireInstanceAccess() middleware. Tampering localStorage only changes nav visibility, not actual access. |
| T-101-15 | Information Disclosure | Sidebar reveals existence of features | mitigate | No-permission tabs are fully hidden (filtered) per D-07. However, an attacker who knows the URL can still access features — server-side auth is the real gate. |
| T-101-16 | Information Disclosure | Stale permissions after role change | accept | Permissions are fetched at login. If a user's roles change mid-session, nav may show stale state until next login. Accept per D-09's silent degradation philosophy — the actual permission check on the backend is always current. |

</threat_model>
