#### Form Validation

| Field | Validation | Error Display |
|-------|-----------|---------------|
| Name | Required, 1-100 chars | Inline `.dialog-error`: "任务名称不能为空" |
| Task description | Required, min 10 chars | Inline `.dialog-error`: "任务内容至少需要10个字符" |
| Cron expression | Must be valid 5-field cron | Inline `.cron-preview-error`: "Cron 表达式无效" |

#### Save Behavior

- **Create (no editingJob):** POST `/api/cron/jobs` with `{ name, description, task_description, cron_expr, enabled }`
- **Edit (editingJob exists):** PUT `/api/cron/jobs/:id` with same fields
- **On success:** Close dialog, reload job list, show toast "保存成功" (create) or "更新成功" (edit)
- **On error:** Show error message in `.dialog-error`, keep dialog open, Save button re-enables
- **Saving state:** Save button shows "保存中..." and is disabled; Cancel stays enabled

### 3. Edit Flow Change (Existing Row Edit)

**Current behavior (Phase 112):** Clicking the edit icon or double-clicking the cron expression opens an inline editor in the table cell.

**New behavior (Phase 113):** Clicking the edit icon opens the task builder dialog with the job's current values pre-filled. The inline cron editor is removed.

- The edit icon button already exists in the `.cell-expr` column (`.cron-edit-btn` with pencil SVG). Its `@click` handler changes from `startEdit(job)` to `openEditDialog(job)`.
- The `@dblclick` handler on cron text also changes to `openEditDialog(job)`.
- The inline editor template (`.cron-editor`, `.cron-presets`, `.cron-preview`) and its state variables (`editingJobId`, `editValue`, `previewDates`, `previewError`) are removed.

### 4. Delete Job (New)

- **Trigger:** Hovering a job row shows a delete icon button (trash SVG) in the Actions column, alongside existing toggle/trigger/logs buttons
- **Confirmation:** Uses same `.dialog-overlay` pattern with delete-specific text:

```
┌──────────────────────────────────────┐
│ 删除任务                             │
│                                      │
│ 确定删除「{job_name}」吗？此操作不可 │
│ 撤销。                               │
│                                      │
│         [取消]  [确认删除]           │
└──────────────────────────────────────┘
```

- **Delete behavior:**
  1. DELETE `/api/cron/jobs/:id`
  2. On success: remove job from local `jobs[]` array (optimistic), show toast "已删除"
  3. On error: show error in dialog, keep dialog open

### 5. Table Column Changes

| Column | Change |
|--------|--------|
| Description (2fr) | No change — still shows `job.description` |
| Handler column | Never existed in the UI table — the `task_description` replaces the backend field but is NOT shown as a table column (too long). It's only visible in the edit/create dialog. |
| Actions column | New delete icon added (shows on row hover, `.btn-icon` style, trash SVG icon). Position after the "日志" button. |

### 6. API Route Integration

| Route | Method | Purpose | UI Trigger |
|-------|--------|---------|------------|
| `/api/cron/jobs` | POST | Create new job | "保存" button in create dialog |
| `/api/cron/jobs/:id` | PUT | Update existing job | "保存" button in edit dialog |
| `/api/cron/jobs/:id` | DELETE | Delete job | Delete confirmation dialog |
| `/api/cron/jobs` | GET | List all jobs | Tab load, after create/update/delete |
| `/api/cron/jobs/:id/toggle` | POST | Toggle enable | Checkbox toggle (no change) |
| `/api/cron/jobs/:id/run` | POST | Manual trigger | "立即执行" button (no change) |
| `/api/cron/jobs/:id/logs` | GET | Get execution logs | Log sub-row expansion (no change) |
| `/api/cron/jobs/preview` | GET | Cron expression preview | Cron input in dialog (existing pattern) |

### 7. Error & Edge Case Handling

| Situation | Treatment |
|-----------|-----------|
| Create API fails (network error) | Error in dialog: "创建失败：{reason}" |
| Update API fails | Error in dialog: "保存失败：{reason}" |
| Delete API fails | Error in dialog: "删除失败：{reason}" |
| Form validation fails before submit | Inline error in dialog, no API call made |
| Name too long | Truncated in table display (existing `text-overflow: ellipsis`) |
| Very long task_description | Textarea handles scroll (`max-height: 240px`, `overflow-y: auto`) |
| Task description contains HTML/JS | Stored as plain text (textarea, not rendered HTML) — no escaping needed |
| cron_jobs table still has `handler` column (pre-migration) | API response still includes it — UI ignores it, treats as optional field |
| Duplicate cron time with existing job | No constraint issue — multiple jobs can share same cron expression |

### 8. Loading & Empty States

| State | Treatment |
|-------|-----------|
| Create dialog initial load | Ready immediately (no data fetch needed) |
| Save in progress | Button text: "保存中..." + disabled, Cancel stays enabled |
| Delete confirmation | No async load needed (uses existing job data) |
| Empty job list (no seed data / all deleted) | Existing "暂无定时任务" message + "新建任务" button prominent |

## Interaction Flow Summary

```
User flow:

[View cron jobs tab]
        │
        ├── [Click "新建任务"]
        │   └── Opens create dialog (empty form)
        │       ├── Fill fields → [保存] → POST /api/cron/jobs → reload
        │       └── [取消] → close
        │
        ├── [Hover row → click edit pencil]
        │   └── Opens edit dialog (pre-filled)
        │       ├── Modify fields → [保存] → PUT /api/cron/jobs/:id → reload
        │       └── [取消] → close
        │
        ├── [Hover row → click delete trash]
        │   └── Opens delete confirmation
        │       ├── [确认删除] → DELETE /api/cron/jobs/:id → reload
        │       └── [取消] → close
        │
        ├── [Toggle switch] → POST /api/cron/jobs/:id/toggle (no change)
        ├── [立即执行] → Trigger dialog + POST /api/cron/jobs/:id/run (no change)

```
