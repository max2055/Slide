## 1. System Classification

**System Type:** Autonomous Task Agent

**Description:**
AI Agent Cron is an NL-driven scheduled task system that replaces 13 hardcoded TypeScript cron handlers with AI agent execution. Users describe tasks in natural language (e.g., "every morning at 9am, check slow queries on production instances and generate a summary report"). At each scheduled tick, the system creates a new agent session, sends the NL task description, and the agent autonomously plans and executes multiple tool calls -- querying databases, analyzing results, generating reports -- then writes the complete execution trace to `cron_job_logs`. The user reviews results through the cron job logs UI.

**Critical Failure Modes:**
1. **Silent execution failure** -- Agent runs but produces incorrect/no results without detection
2. **Infinite tool-call loops** -- Agent gets stuck calling tools repeatedly without converging
3. **Stale/incorrect data** -- Agent acts on cached or wrong database state
4. **Timeout without partial results** -- 5-minute timeout kills the agent with no trace of partial work
5. **Permission escalation** -- Agent accesses databases or performs operations outside the cron job's scope

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Database Operations / DevOps (DBOps)

**User Population:** Database Administrators (DBAs), site reliability engineers, operations engineers managing database instances (MySQL, Oracle, MongoDB) in a multi-instance production environment. Users are technical operations practitioners who currently rely on hardcoded scripts and dashboard monitoring.

**Stakes Level:** Medium (operational)

**Output Consequence:** The DBA reads the agent's analysis (slow query report, health summary, RCA) and makes an operational decision: tune a query, allocate more storage, restart an instance, or take no action. A missed degradation can lead to cascading performance issues affecting downstream applications. A false alert wastes DBA time and erodes trust in the system. The cron agent does not execute destructive operations -- it observes, analyzes, and reports.

### What Domain Experts Evaluate Against

```
Dimension: Problem Detection Accuracy (Signal vs. Noise)
Good (DBA would accept): Agent correctly identifies a real performance degradation with specific,
  attributable evidence -- "Instance PROD-DB-01: avg query latency increased from 12ms to 47ms
  over the last 15 minutes, driven by a full table scan on orders.status by session 4271."
Bad (DBA would flag): Agent reports "CPU spike detected" during a known nightly ETL window when
  high CPU is normal, or it fails to detect a slow-query buildup because each individual query
  was just under the threshold.
Stakes: High
Source: Production DBA practice -- false negatives cause undetected degradation; false positives
  cause alert fatigue and system abandonment (research shows 60-99% of static-threshold alerts
  are non-actionable).

---

Dimension: Analysis Actionability (So What / Now What)
Good (DBA would accept): Report tells the DBA what changed, why it matters, and what to investigate
  -- "orders_pkey scan on orders.status = 'PENDING' suggests a missing composite index on
  (status, created_at). Estimate 73% of the 12,000 full-scan rows could be eliminated."
Bad (DBA would flag): Report dumps raw metric values without interpretation ("CPU=82%,
  IOPS=3400, connections=127"), or states a vague conclusion ("performance is degraded")
  with no specifics to act on.
Stakes: High
Source: DBA incident response workflow -- a non-actionable alert adds cognitive load and delays
  remediation; an actionable one cuts MTTR.

---

Dimension: Temporal Baseline Awareness
Good (DBA would accept): Agent compares current metrics against learned baselines for the same
  time-of-day and day-of-week, and accounts for known workload patterns. Example: "CPU 78% is
  within the normal range for this instance on weekdays at 2 AM (baseline 65-85%)."
Bad (DBA would flag): Agent uses a static threshold (e.g., "CPU > 80% = problem") that triggers
  on expected batch-job activity, or it computes a baseline that includes an anomaly period
  and treats the anomaly as normal.
Stakes: Critical
Source: Industry research on adaptive baselines -- static thresholds produce 60-80% false positives;
  DBA practice relies on per-instance, per-time-of-day baselines for meaningful anomaly detection.

---

Dimension: Cross-Metric Correlation (Compound Reasoning)
Good (DBA would accept): Agent correlates multiple metrics before drawing a conclusion -- "High I/O
  wait (85%) + increased query latency (3x) + tempdb contention = concurrent index rebuild is
  impacting production queries. Consider scheduling during maintenance window."
Bad (DBA would flag): Agent raises concern based on a single out-of-range metric without checking
  related symptoms -- flags "disk I/O 95%" without checking whether this is caused by a
  known backup job or whether it is actually impacting query throughput.
Stakes: Medium
Source: Database observability best practices (SolarWinds compound alerting, DBA Dash
  multi-condition alerting) -- single-metric alerts are the leading cause of non-actionable noise.

---

Dimension: Execution Completeness and Traceability
Good (DBA would accept): Every cron tick produces a self-contained execution trace with timestamps,
  each tool call and its result, the agent's reasoning at each decision point, and a final
  conclusion. A DBA reviewing the log can reconstruct what happened and why.
Bad (DBA would flag): Agent silently fails (marks task "success" with empty or boilerplate
  content), or times out with no partial results and no indication of how far the analysis got.
Stakes: High
Source: Production operations standard -- cron job logs are the primary audit artifact; incomplete
  or misleading logs are worse than no logs because they create false confidence.
```

### Known Failure Modes in This Domain

1. **False negative on slow degradation** -- The agent checks metrics at a snapshot in time and concludes "healthy" while a gradual trend (e.g., buffer cache hit ratio declining 0.5% per day over a week) goes undetected. A DBA looking at a 7-day trend chart would catch this immediately.

2. **Alert fatigue from over-reporting** -- The agent flags every metric fluctuation as noteworthy. If the system generates 40 reports per day and only 1-2 are actionable, DBAs learn to ignore it. Research shows that after 30 days of high false-positive rates, DBA response time degrades 2-3x.

3. **Outdated baseline poisoning** -- The agent uses a baseline computed during an anomalous period (e.g., a migration week), so it treats abnormal metrics as "normal." Subsequent true anomalies fall within the poisoned baseline range and are never flagged.

4. **Context fragmentation across multi-step analysis** -- In a task like "check all instances for slow queries and generate a summary," the agent may detect a slow query in step 3 but fail to connect it to the high I/O wait found in step 5, producing a report that misses the root cause.

5. **Silent timeout without partial output** -- The 5-minute timeout kills the agent mid-analysis. The cron job log shows `status: timeout` with no indication of what was collected or analyzed before the cutoff. The DBA has no way to determine whether the partial data is actionable.

### Regulatory / Compliance Context

This is an internal database operations automation tool. No direct regulatory requirements (HIPAA, GDPR, SOX) govern the cron system itself. However, the agent executes queries against production databases that may contain sensitive business data. The following constraints apply:

- **Access scope**: Agent must only access databases and instances authorized by the cron job's configuration -- not all databases reachable by the platform.
- **No destructive execution**: The agent has `db_sql_execute` capability. Cron tasks must be configured as read-only unless explicitly authorized. The system must prevent cron agents from running DDL/DML.
- **Audit trail**: Every execution must produce a complete, non-repudiable log. The `cron_job_logs` table and agent session trace together form the audit record.
- **Data retention**: Agent traces contain query samples (slow query text, schema snapshots). These are operational data, not customer PII, but they should not be retained longer than the platform's data retention policy (default 30 days).

## 2. Framework Decision

**Selected Framework:** `@slide/agent-core` (TypeScript port of nanobot)

**Version:** Current workspace package (`packages/agent-core/`)

**Rationale:**
Phase 108-111 already migrated the project from OpenClaw Gateway to `@slide/agent-core` -- a TypeScript port of the nanobot agent framework. This is the project's own agent infrastructure, providing `AgentRunner` (LLM-tool execution loop), `ToolRegistry`, `SessionManager` (JSONL-persisted), `LLMProvider` (Anthropic/OpenAI/Ollama), and a minimal WebSocket transport. No new framework is needed because:

1. **Already integrated** -- `DirectAdapter` wraps `AgentRunner` behind the `IAgentEngine` interface used by all platform code
2. **Model-agnostic** -- Supports Anthropic, OpenAI, and Ollama through a pluggable `LLMProvider`
3. **Tool-rich** -- Existing tool catalog includes database health checks, slow query retrieval, Oracle ASH/AWR reports, etc. -- exactly what cron tasks need
4. **Session isolation** -- `SessionManager` creates independent JSONL sessions per sessionKey, giving each cron execution a clean context
5. **Observability built-in** -- `AgentRunner` emits hooks (`beforeExecuteTools`, `afterIteration`, `afterComplete`) that can be wired to `cron_job_logs`

The existing `IAgentEngine.invoke()` (fire-and-forget) needs extension to support result collection for cron -- but this is an adapter extension, not a framework swap.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| OpenClaw Gateway (legacy) | Already migrated away in Phase 110/111; deprecated in this project |
| Claude Agent SDK | Adds new dependency; @slide/agent-core already provides the agent loop |
| LangGraph | Python-only for advanced features; overkill for scheduled task execution with linear flow |
| CrewAI | Python-only; multi-agent orchestration not needed for single-task execution |
| OpenAI Agents SDK | Vendor lock-in; project needs model-agnostic (Anthropic + OpenAI + Ollama) |

**Vendor Lock-In Accepted:** No -- LLMProvider abstraction supports Anthropic, OpenAI, and Ollama interchangeably.

### Entry Point Pattern -- Cron Invoke-with-Result

The existing `IAgentEngine.invoke()` is **fire-and-forget** -- it returns only `{ content, usage }` and discards the execution trace (tool names, events, partial results). For cron, we need the full `AgentRunResult`. Two approaches:

**Approach A (Recommended): Extend `IAgentEngine` with `execute()`**

Add a new method to `IAgentEngine`:

```typescript
// In apps/db-ops-api/src/adapter/types.ts
import type { AgentRunResult } from '@slide/agent-core';

export interface IAgentEngine {
  // ... existing start(), chat(), invoke(), listTools(), capabilities()

  /**
   * Full execution with complete result trace.
   * Unlike invoke() (fire-and-forget), this returns the full AgentRunResult
   * including toolEvents, toolsUsed, all messages, and usage.
   * Used by cron to capture complete execution trace.
   */
  execute(
    sessionKey: string,
    message: string,
    systemPrompt?: string,
  ): Promise<AgentRunResult>;
}
```

Implementation in `DirectAdapter`:

```typescript
// In apps/db-ops-api/src/adapter/direct-adapter.ts

async execute(
  sessionKey: string,
  message: string,
  systemPrompt?: string,
): Promise<AgentRunResult> {
  const session = this.sessionManager.getOrCreate(sessionKey);

  // Build messages with ContextBuilder
  const ctxMessages = systemPrompt
    ? [
        { role: 'system', content: systemPrompt } as Message,
        { role: 'user', content: message } as Message,
      ]
    : (await this.contextBuilder.buildMessages([], message)) as Message[];

  // Persist user message
  session.addMessage('user', message);

  // Collect tool events for the execution trace
  const collectedEvents: ToolEvent[] = [];
  const hook = new (class extends NoopHook {
    override async afterIteration(ctx: AgentHookContext) {
      collectedEvents.push(...ctx.toolEvents);
    }
  })();

  try {
    const result = await this.runner.run({
      initialMessages: ctxMessages,
      tools: this.registry,
      model: this.provider.getDefaultModel(),
      maxIterations: 20,
      maxToolResultChars: 20000,
      temperature: 0.0,
      reasoningEffort: 'medium',
      hook,
      contextWindowTokens: 200_000,
      maxTokens: 4096,
      llmTimeoutS: 300,        // 5 min per D-06
      failOnToolError: false,   // allow the agent to retry failed tools
      sessionKey,
    });

    // Persist assistant response
    if (result.finalContent) {
      session.addMessage('assistant', result.finalContent);
    }
    await this.sessionManager.save(session);

    return result;
  } catch (err) {
    const errorMessage = err instanceof Error ? err.message : String(err);
    // Save session even on error for audit trail
    try { await this.sessionManager.save(session); } catch { /* best-effort */ }
    throw err;
  }
}
```

### Key Abstractions

| Abstraction | File | Purpose in Cron Context |
|-------------|------|------------------------|
| `AgentRunner` | `runner.ts` | LLM-tool execution loop. Runs up to `maxIterations` turns. Handles context governance (microcompact, snip, budget enforcement) and parallel tool execution automatically. |
| `ToolRegistry` | `tool-registry.ts` | Registers and validates tools. `execute(name, params)` runs a tool with JSON Schema validation + param casting. `getDefinitions()` returns tool schemas for LLM function declarations. |
| `Session` + `SessionManager` | `session.ts` | Session stores message history per conversation key. `SessionManager` persists sessions as JSONL files under `.slide/sessions/`. Cron creates one session per tick (key: `cron:<jobId>:<timestamp>`). |
| `LLMProvider` | `types.ts` (interface) | Pluggable provider abstraction. Two implementations: `AnthropicProvider` (env var: `ANTHROPIC_API_KEY` + `ANTHROPIC_MODEL`) and `OpenAIProvider` (constructor: `apiKey`, `baseURL`, `model`). Default model for Anthropic: `claude-sonnet-4-20250929`. |
| `AgentHook` | `types.ts` | Lifecycle hook interface. `NoopHook` = no-op. For cron: extend `NoopHook` and override `afterIteration()` to collect `ToolEvent[]` for the execution log. |
| `ContextBuilder` | `context.ts` | Assembles system prompt from bootstrap files (`SOUL.md`, `AGENTS.md`, `HEARTBEAT.md`), `MemoryStore` context, `SkillsLoader` summaries, and runtime info (current time). Cron tasks may skip this and use a custom system prompt instead. |

### Common Pitfalls

1. **invoke() discards execution trace.** Current `DirectAdapter.invoke()` returns only `{ content, usage }`. Tool names, per-tool status, and intermediate messages are lost. Cron must use `AgentRunner.run()` directly or extend `IAgentEngine.execute()` to get the full `AgentRunResult` for `cron_job_logs`.

2. **failOnToolError = true crashes the agent.** Default is not set explicitly in existing invoke patterns. If a DB query tool fails, the agent should try a different query, not crash. Always set `failOnToolError: false` and let the agent's error-recovery prompt handle it. The runner itself has guards against infinite loops (`repeatedExternalLookupError` at 5+ calls, `MAX_EMPTY_RETRIES` = 2, `MAX_LENGTH_RECOVERIES` = 3).

3. **maxIterations = 5 is too few for multi-step cron tasks.** Existing `invoke()` uses `maxIterations: 5`. Multi-step cron tasks (e.g., check health -> find slow queries -> analyze -> generate report) need 15-20 iterations. The runner's guardrails (pending queue, injection cycles) prevent runaway loops, so higher limits are safe.

4. **contextWindowTokens must be set explicitly.** The `AgentRunner` applies `snipHistory()` only when `contextWindowTokens` is set in the run spec. If omitted, there is no upper bound on context growth, and the agent will eventually hit the LLM provider's max tokens. Always set `contextWindowTokens: 200_000` and `maxToolResultChars` to 10K-20K.

5. **Duplicate sessionKey causes interleaved history.** If two cron ticks share the same `sessionKey`, both write to the same JSONL file and message history becomes interleaved. Always include the job ID + execution timestamp in the key (e.g., `cron:<jobId>:<unixTimestampMs>`). Clean up old sessions via `SessionManager.deleteSession()` after the log is written.

6. **NoopHook loses all tool events for logging.** If `hook: new NoopHook()` is used, `AgentHookContext.toolEvents` is never captured because `NoopHook.afterIteration()` is empty. For observability, extend `NoopHook` and override `afterIteration()` to collect events into an array.

## 4. Implementation Guidance

**Model Configuration:**

Default model (from `AnthropicProvider`): `claude-sonnet-4-20250929`. Override via `ANTHROPIC_MODEL` env var or via DB LLM provider configuration in `getAgentEngine()`.

Recommended params for cron tasks:

```typescript
model: provider.getDefaultModel(),       // claude-sonnet-4-20250929 or as configured
temperature: 0.0,                        // deterministic output for scheduled tasks
reasoningEffort: 'medium',               // extended thinking for multi-step planning
maxTokens: 4096,                         // enough for a summary report
maxIterations: 20,                       // multi-step: health check, queries, analysis, report
llmTimeoutS: 300,                        // 5 min wall-clock timeout per D-06
```

For OpenAI-compatible providers (via `OpenAIProvider`), set `reasoningEffort` only for providers that support it (e.g., DeepSeek). The `OpenAIProvider` passes `reasoning_effort` through successfully for compatible APIs.

**Core Pattern -- Cron Execution with Result Collection:**

```typescript
import { AgentRunner, NoopHook } from '@slide/agent-core';
import type { AgentHook, AgentHookContext, ToolEvent } from '@slide/agent-core';

// 1. Create a custom hook to collect execution trace
class CronHook extends NoopHook {
  public events: ToolEvent[] = [];

  override async afterIteration(ctx: AgentHookContext) {
    this.events.push(...ctx.toolEvents);
  }
}

// 2. Build the system prompt for the cron task
function buildCronSystemPrompt(taskDescription: string): string {
  return `You are a database operations automation agent. Your task:

${taskDescription}

Requirements:
- Execute database tools (db_health_check, db_slow_queries, db_performance_analysis, db_sql_execute) to gather data
- Analyze the results and determine next steps autonomously
- If a tool errors, retry with different parameters or try an alternative approach
- When the task is complete, call slide_complete_cron with:
  - status: "success" | "failure" | "partial"
  - summary: a concise report of findings
  - details: the full analysis data
- Maximum execution time: 5 minutes

Available tools provide database health checking, slow query analysis, performance diagnostics,
SQL execution, and report generation. Use them in sequence as needed.`;
}

// 3. Execute the cron task
async function executeCronTask(
  jobId: number,
  taskDescription: string,
  provider: LLMProvider,
  registry: ToolRegistry,
): Promise<AgentRunResult> {
  const runner = new AgentRunner(provider);
  const sessionKey = `cron:${jobId}:${Date.now()}`;
  const hook = new CronHook();

  const result = await runner.run({
    initialMessages: [
      { role: 'system', content: buildCronSystemPrompt(taskDescription) },
      { role: 'user', content: taskDescription },
    ],
    tools: registry,
    model: provider.getDefaultModel(),
    maxIterations: 20,
    maxToolResultChars: 20000,
    temperature: 0.0,
    reasoningEffort: 'medium',
    hook,
    contextWindowTokens: 200_000,
    maxTokens: 4096,
    llmTimeoutS: 300,
    failOnToolError: false,
    sessionKey,
  });

  return result;
}

// 4. Log the result
async function logCronResult(
  jobId: number,
  result: AgentRunResult,
): Promise<void> {
  const status = result.error
    ? 'error'
    : result.finalContent?.toLowerCase().includes('success')
      ? 'success'
      : 'completed';

  await cronJobLogDatabaseService.addLog({
    job_id: jobId,
    status,
    result: result.finalContent,
    tools_used: result.toolsUsed,
    tool_events: result.toolEvents,
    usage: result.usage,
    stop_reason: result.stopReason,
    executed_at: new Date(),
  });
}
```

**Tool Use:**

The cron agent has access to the full Slide platform tool catalog (auto-discovered via `getAgentEngine()` -> `loadPlatformTools()`). Key tool categories:

| Category | Tool Names | Cron Use Case |
|----------|-----------|---------------|
| Health check | `db_health_check` | Verify instance connectivity and status |
| Slow query | `db_slow_queries` | Gather slow queries for analysis |
| Performance | `db_performance_analysis` | Get performance metrics |
| SQL explain | `db_sql_explain` | Analyze query plans |
| SQL execute | `db_sql_execute` | Run diagnostic queries |
| Oracle ASH/AWR | `db_ash_report`, `db_awr_report` | Oracle-specific diagnostics |
| Cron completion | `slide_complete_cron` | Save structured result to cron_job_logs |

The `slide_complete_cron` tool (analogous to the existing `slide_complete_analysis` at `tools/generated/slide-self-mgmt/complete_analysis.ts`) is the final tool the agent calls to persist results. It accepts:

```typescript
interface CompleteCronParams {
  jobId: number;
  status: 'success' | 'failure' | 'partial';
  summary: string;
  details: Record<string, unknown>;
}
```

Parallel tool execution: Tools with `concurrencySafe: true` are batched and executed concurrently via `Promise.all()` in `AgentRunner`. Use this for independent diagnostic queries (e.g., `db_health_check` on multiple instances). The runner uses Kahn's algorithm for dependency-aware batching.

**Session Isolation Strategy:**

Each cron tick gets its own session with a unique key:

```
cron:<jobId>:<unixTimestampMs>
```

Example: `cron:42:1716793200000`

Isolation guarantees:
- `SessionManager.getOrCreate()` creates a new `Session` if the key does not exist
- The session is persisted to `.slide/sessions/<sha256>.jsonl` -- isolated per key
- After the result is logged to `cron_job_logs`, the session can be cleaned up via `SessionManager.deleteSession(sessionKey)`
- For audit trail: keep the session until the next successful execution, then rotate

Session lifecycle for a single cron tick:

```
1. sessionKey = `cron:<jobId>:${Date.now()}`
2. session = sessionManager.getOrCreate(sessionKey)
3. runner.run({ ..., sessionKey })
4. cronJobLogDatabaseService.addLog({ ... })
5. sessionManager.save(session)       // persisted for audit
6. sessionManager.deleteSession()     // optional cleanup
```

**Context Window Strategy:**

The `AgentRunner` provides four built-in context governance mechanisms, applied in order before every LLM call:

1. **`microcompact()`** -- Compacts tool results for known high-volume tools (members of `COMPACTABLE_TOOLS`: `read_file`, `exec`, `grep`, `find_files`, `web_search`, `web_fetch`, `list_dir`, `list_exec_sessions`). Replaces stale results (older than the most recent 10) with `"[tool result omitted from context]"`. This prevents repetitive large tool outputs from eating context space.

2. **`applyToolResultBudget()`** -- Truncates each tool result to `maxToolResultChars` (recommended: 20000). Prevents any single tool output from dominating the context window.

3. **`snipHistory()`** -- When `contextWindowTokens` is set, estimates token usage for the full message chain and drops the oldest non-system messages if over budget. Keeps the most recent messages up to `contextWindowTokens - maxTokens - SNIP_SAFETY_BUFFER`. System prompts are never dropped.

4. **`dropOrphanToolResults()` / `backfillMissingToolResults()`** -- Removes tool results whose corresponding tool call no longer exists in the message chain, and backfills missing results with `"[Tool result unavailable]"`. Ensures message structure is always valid for the provider.

5-minute cron tasks typically produce 5-15 tool calls across 1-3 planning iterations. With `contextWindowTokens: 200_000` and `maxToolResultChars: 20000`, context pressure is extremely low. The governance mechanisms are safety nets, not primary design concerns, for cron tasks.

For unusually long-running tasks (approaching 5 min), set `contextBlockLimit` to reserve more output tokens:

```typescript
contextWindowTokens: 200_000,
contextBlockLimit: 180_000,  // reserve 20K tokens for output
```

### Structured Outputs with Zod (Not Pydantic -- This Is TypeScript)

The cron agent must produce structured results for `cron_job_logs`. Use Zod schemas for runtime validation.

**Define the output schema:**

```typescript
import { z } from 'zod';

// Schema for the final cron completion payload
export const CronCompletionSchema = z.object({
  status: z.enum(['success', 'failure', 'partial']),
  summary: z.string().min(1).max(5000),
  details: z.record(z.unknown()),
  metrics: z
    .object({
      databasesChecked: z.number().int().optional(),
      slowQueriesFound: z.number().int().optional(),
      anomaliesDetected: z.number().int().optional(),
      actionsTaken: z.array(z.string()).optional(),
    })
    .optional(),
  errors: z
    .array(
      z.object({
        tool: z.string(),
        message: z.string(),
        recovered: z.boolean(),
      }),
    )
    .optional(),
});

export type CronCompletion = z.infer<typeof CronCompletionSchema>;
```

**Integration with `@slide/agent-core`:**

The framework uses JSON Schema for tool parameter validation (not Zod). The `ToolRegistry.prepareCall()` validates parameters using `validateJsonSchema()` -- a custom JSON Schema validator in `tool-registry.ts`. Zod is used at the **application layer** to validate the structured output from the LLM.

Pattern: Define a completion tool with JSON Schema parameters for the AgentRunner, then validate the LLM's output against the Zod schema:

```typescript
// Tool definition with JSON Schema (what the LLM sees)
const completeCronTool: Tool = {
  name: 'slide_complete_cron',
  description: 'Complete the cron task and save results to the log. Call this when the task is finished.',
  parameters: {
    type: 'object',
    properties: {
      status: { type: 'string', enum: ['success', 'failure', 'partial'] },
      summary: { type: 'string', description: 'Concise report of findings' },
      details: { type: 'object', description: 'Full analysis data' },
    },
    required: ['status', 'summary'],
  },
  readOnly: true,
  concurrencySafe: false,
  exclusive: false,
  execute: async (params: Record<string, unknown>) => {
    // Validate LLM output against Zod schema before persisting
    const parsed = CronCompletionSchema.safeParse(params);
    if (!parsed.success) {
      return `Invalid completion data: ${parsed.error.message}. Please correct and retry.`;
    }
    // Persist to cron_job_logs
    await cronJobLogDatabaseService.updateResult(
      parsed.data.status,
      parsed.data.summary,
      parsed.data.details,
    );
    return `Task completed with status: ${parsed.data.status}`;
  },
};
```

**Retry logic:** If Zod validation fails, the tool returns an error message with the validation details. The LLM receives this as a tool result error and retries with corrected data. The runner has built-in `repeatedExternalLookupError` guard at 5+ retries of the same tool -- so validation failures surface as task errors within 5 attempts.

```typescript
// In the tool execute function, for consistent error format:
execute: async (params) => {
  const parsed = CronCompletionSchema.safeParse(params);
  if (!parsed.success) {
    // Return structured error for LLM to fix (not throw)
    return `Validation error: ${parsed.error.issues.map(i =>
      `${i.path.join('.')}: ${i.message}`
    ).join('; ')}`;
  }
  // ... persist ...
}
```

### Async-First Design

`@slide/agent-core` is fully async -- `AgentRunner.run()`, `ToolRegistry.execute()`, `SessionManager.save()`, and all hook methods return `Promise`.

**Pattern for cron execution in a Fastify handler:**

```typescript
import type { FastifyInstance } from 'fastify';

export async function registerCronRoutes(app: FastifyInstance): Promise<void> {
  app.post('/api/cron/:jobId/trigger', async (request, reply) => {
    const { jobId } = request.params as { jobId: string };

      parseInt(jobId, 10),

    });

```

**One common mistake: running AgentRunner inside `setTimeout` without proper error handling.**

```typescript
// WRONG -- unhandled promise rejection
setTimeout(() => {
  agentRunner.run(spec);   // Promise is ignored
}, 5000);

// CORRECT -- proper async with error handling
setTimeout(async () => {
  try {
    const result = await agentRunner.run(spec);
    await logCronResult(jobId, result);
  } catch (err) {
    await logCronError(jobId, err);
  }
}, 5000);
```

**Stream vs. await for cron:**

Cron tasks use `await` (non-streaming) because:
- Results are written to logs, not displayed to a waiting user
- Streaming overhead is unnecessary for backend-only execution
- The hook should NOT have `wantsStreaming()` return `true`

Use streaming only when the result feeds a real-time UI (the existing `DirectAdapter.chat()` pattern). For cron, use `NoopHook` or a minimal extension:

```typescript
const hook = new (class extends NoopHook {
  override async afterIteration(ctx: AgentHookContext) {
    collectedEvents.push(...ctx.toolEvents);
  }
})();
```

### Prompt Engineering Discipline

**System vs. user prompt separation:**

The `@slide/agent-core` `Message` type supports `role: "system"` and `role: "user"`. The `AnthropicProvider` extracts system messages from the array and passes them via the Anthropic `system` parameter (which has a higher instruction-following weight). The `OpenAIProvider` passes them inline.

For cron, the system prompt should contain:

1. **Role definition** -- "You are a database operations automation agent."
2. **Task description** -- The NL task from `cron_jobs.task_description`
3. **Constraints** -- Time limit, error recovery rules, tool usage guidelines
4. **Completion protocol** -- Which tool to call when done, what data to include

The user message can be minimal (repeat the task description) or omitted entirely if the system prompt is self-contained.

```typescript
const SYSTEM_PROMPT_TEMPLATE = `You are a database operations automation agent.

TASK: {{taskDescription}}

CONSTRAINTS:
- Maximum execution time: 5 minutes
- If a database query fails, retry with different parameters
- Use multiple tools in sequence: gather data, analyze, then report
- When the task is complete, call slide_complete_cron with the results
- Do NOT ask for confirmation -- execute autonomously

WORKFLOW PATTERN:
1. First, check system health with db_health_check
2. Based on health results, run specific diagnostic queries
3. Analyze the collected data
4. Generate a summary report
5. Call slide_complete_cron to persist results`;
```

**Few-shot examples:**

For cron tasks, embed the expected output format directly in the system prompt rather than using few-shot examples. The task description is already NL and the agent is role-instructed. Few-shot examples are useful only when the output format is highly specific (e.g., a structured JSON report). In that case, inject the examples dynamically based on the task type:

```typescript
function buildSystemPrompt(task: CronJob): string {
  const examples = task.outputFormat === 'json'
    ? '\n\nExample output format:\n```json\n{\n  "status": "success",\n  "databases_checked": 5,\n  "slow_queries_found": [...],\n  "summary": "...",\n  "recommendations": [...]\n}\n```'
    : '';

  return SYSTEM_PROMPT_TEMPLATE.replace('{{taskDescription}}', task.task_description) + examples;
}
```

**Set `maxTokens` explicitly, never leave unbounded:**

```typescript
// ALWAYS set maxTokens -- even for cron
maxTokens: 4096,   // sufficient for a diagnostic summary report
```

The `AgentRunner` has a `length` finish reason handler (`MAX_LENGTH_RECOVERIES = 3`) that prompts the LLM to continue if `maxTokens` was hit. This is safe, but the recovery message adds an extra turn, consuming more budget.

### Context Window Management

**RAG (not applicable to this Autonomous Task Agent system):**

This system does not use RAG. Context is the task description + tool results + conversation history. The context governance mechanisms listed in Section 4 are the primary management strategy.

**Multi-step autonomous task summarization:**

As the agent executes multiple tool calls, the conversation grows. The runner automatically applies `microcompact()` to compress stale tool results for known compactable tools. For cron-specific tools (database diagnostic tools), add them to `COMPACTABLE_TOOLS` if they produce large outputs:

```typescript
// In packages/agent-core/src/runner.ts (extend for cron)
const COMPACTABLE_TOOLS.add('db_slow_queries');
const COMPACTABLE_TOOLS.add('db_performance_analysis');
const COMPACTABLE_TOOLS.add('db_ash_report');
const COMPACTABLE_TOOLS.add('db_awr_report');
```

**5-minute budget strategy:**

Cron tasks are short (5 min max). With `maxIterations: 20`, each iteration has ~15 seconds for the LLM call + tool execution. The context rarely exceeds 50K tokens for a typical 3-5 turn execution. The built-in `snipHistory()` at 200K tokens is a safety net, not operational.

For longer-running tasks that approach the 5-minute limit, the runner's `llmTimeoutS` will terminate the LLM call. The `checkpointCallback` (if set) persists partial state, allowing a retry mechanism to resume from where it left off.

```typescript
// For resumable cron tasks: save checkpoint every turn
checkpointCallback: async (payload) => {
  session.metadata['runtime_checkpoint'] = payload;
  await sessionManager.save(session);
},
```

### Cost and Latency Budget

**Per-call cost estimate:**

Model: `claude-sonnet-4-20250929`

| Resource | Estimate |
|----------|----------|
| Input tokens per task | ~500 (system) + ~200 (user) + ~5000-15000 (tool results) = ~5K-16K |
| Output tokens per task | ~200 (reasoning) + ~2000-4000 (final report) = ~2K-4K |
| Turns per task | 3-5 (plan, execute tools, report) |
| Total input tokens | ~15K-80K per execution |
| Total output tokens | ~6K-20K per execution |
| Cost per execution (Sonnet) | ~$0.10-$0.50 |
| Cost per execution (Haiku, if suitable for simple tasks) | ~$0.01-$0.05 |

**At 13 cron jobs x 1 execution each = ~$1.30-$6.50/day** (Sonnet) or ~$0.13-$0.65/day (Haiku).

**Caching strategy:**

The `@slide/agent-core` runtime does not include built-in caching. Implement two layers:

1. **Exact-match cache** -- For tasks that produce the same result every tick (e.g., "check if instance is healthy"). Use the task description + timestamp bucket (e.g., same hour) as cache key. Skip execution entirely if the cache is fresh.

2. **Semantic cache** -- For result retrieval. When a cron task produces the same output as the previous execution (e.g., "all instances healthy"), skip logging a redundant result. Compare the new summary against the previous log entry's summary via string similarity (Jaccard or Levenshtein).

```typescript
// Simple exact-match cache for cron
const cronResultCache = new Map<string, { result: string; timestamp: number }>();
const CACHE_TTL = 60 * 60 * 1000; // 1 hour

async function getCachedOrExecute(jobId: number, taskDescription: string, provider: LLMProvider, registry: ToolRegistry): Promise<AgentRunResult> {
  const cacheKey = `cron:${jobId}:${taskDescription}`;
  const cached = cronResultCache.get(cacheKey);
  if (cached && Date.now() - cached.timestamp < CACHE_TTL) {
    return cached as unknown as AgentRunResult; // Return cached result
  }
  const result = await executeCronTask(jobId, taskDescription, provider, registry);
  cronResultCache.set(cacheKey, { result: result.finalContent ?? '', timestamp: Date.now() });
  return result;
}
```

**Cheaper models for sub-tasks:**

If the task requires classification or routing (e.g., "determine whether this is a performance issue or an availability issue", or "summarize collected data"), use a separate `AgentRunner` instance with a cheaper model:

```typescript
// Classification sub-task with smaller/cheaper model
const classifierProvider = new OpenAIProvider({
  apiKey: process.env.OPENAI_API_KEY!,
  model: 'gpt-4o-mini',  // cheaper than Sonnet for simple decisions
});

async function classifyIssue(dbData: string): Promise<string> {
  const runner = new AgentRunner(classifierProvider);
  const result = await runner.run({
    initialMessages: [
      { role: 'system', content: 'Classify the database issue into one of: performance | availability | security | other. Respond with only the category.' },
      { role: 'user', content: dbData },
    ],
    tools: new ToolRegistry(),
    model: 'gpt-4o-mini',
    maxIterations: 1,
    maxToolResultChars: 1000,
    temperature: 0.0,
    hook: new NoopHook(),
    maxTokens: 50,  // just a single word
  });
  return result.finalContent?.trim().toLowerCase() ?? 'other';
}
```

## 5. Evaluation Strategy

This is an **internal DBA operations tool**, not customer-facing. The eval strategy prioritizes operational correctness and trace quality over subjective quality. Prefer simple code-based checks over complex LLM-judge setups.

### 5.1 Evaluation Dimensions

Six dimensions: five from domain expert rubric ingredients (Section 1b) plus one cross-cutting Operational Safety dimension.

| # | Dimension | Priority | PASS (DBA would accept) | FAIL (DBA would flag) | Measurement |
|---|-----------|----------|------------------------|----------------------|-------------|
| D1 | **Problem Detection Accuracy** (Signal vs. Noise) | High | Agent correctly identifies a real performance degradation with specific, attributable evidence -- "Instance PROD-DB-01: avg query latency increased from 12ms to 47ms over the last 15 minutes, driven by a full table scan on orders.status by session 4271." | Agent reports "CPU spike detected" during known nightly ETL window, or fails to detect slow-query buildup because each individual query was just under threshold. | Code + Human |
| D2 | **Analysis Actionability** (So What / Now What) | High | Report tells DBA what changed, why it matters, and what to investigate -- "orders_pkey scan on orders.status = 'PENDING' suggests a missing composite index on (status, created_at)." | Report dumps raw metric values without interpretation ("CPU=82%, IOPS=3400"), or states vague conclusion with no specifics to act on. | LLM Judge + Human |
| D3 | **Temporal Baseline Awareness** | Critical | Agent compares current metrics against learned baselines for same time-of-day and day-of-week, accounts for known workload patterns. | Agent uses static threshold ("CPU > 80% = problem") triggering on expected batch-job activity, or computes baseline that includes anomaly period and treats anomaly as normal. | Code |
| D4 | **Cross-Metric Correlation** (Compound Reasoning) | Medium | Agent correlates multiple metrics before drawing conclusion -- "High I/O wait (85%) + increased query latency (3x) + tempdb contention = concurrent index rebuild impacting production." | Agent raises concern based on single out-of-range metric without checking related symptoms. | Code |
| D5 | **Execution Completeness and Traceability** | High | Every cron tick produces self-contained trace with timestamps, each tool call and its result, agent reasoning at each decision point, and a final conclusion. | Agent silently fails (marks "success" with empty/boilerplate content), or times out with no partial results and no indication of progress. | Code |
| D6 | **Operational Safety** | Critical | Agent only queries read-only data; never attempts DDL/DML; respects cron job's database scope. | Agent attempts DDL/DML or queries unauthorized databases/instances. | Code |

**Measurement approach per dimension:**

- **Code** (D1, D3, D4, D5, D6): Deterministic validation requiring no LLM judgment. Includes: Zod schema validation of `slide_complete_cron` payload, presence/absence checks on required fields, timeout detection via `stopReason`, tool-call count analysis, parse of SQL statements in `db_sql_execute` calls for destructive keywords (DROP, ALTER, TRUNCATE, DELETE, INSERT, UPDATE without SELECT-wrapping), check that `db_health_check` was called before diagnostic queries.

- **LLM Judge** (D2 only): An LLM (separate model call) scores the agent's final report against a rubric for actionability. Steps: (1) extract the `summary` and `details` fields from `cron_job_logs`, (2) prompt the judge with the rubrics from Section 1b, (3) score 1-5 on actionability. Requires calibration against DBA judgments before trusting (target: >= 0.7 correlation, per `ai-evals.md`). Run on sampled logs, not every execution.

- **Human** (D1, D2): DBA reviews sampled execution traces for false positives, false negatives, and actionability. Run at eval dataset creation time for calibration, then periodic production sampling (weekly, 5% of traces).

## 6. Guardrails

Guardrail vs. flywheel decision (per `ai-evals.md`): ask "If this behavior goes wrong, would it be catastrophic?" For an internal DBA tool, "catastrophic" means: data loss, undetected production degradation, or loss of audit trail.

### 6.1 Online Guardrails (Real-Time)

Runs on every cron execution. Each adds latency -- keep minimal.

| Guardrail | Failure Mode Addressed | Mechanism | Implementation |
|-----------|----------------------|-----------|----------------|
| **Timeout enforcement** | CFM #4 (timeout without partial results) | Wall-clock timer at `llmTimeoutS: 300`. If exceeded, force-kill the AgentRunner and write partial trace to `cron_job_logs` with status `timeout`. | `AgentRunner.run()` accepts `llmTimeoutS` param. Wrap the run call in `Promise.race()` with a setTimeout that triggers `session.save()` + `cronJobLogService.addLog()` with whatever `CronHook.events` has collected so far. |
| **Tool-call loop detection** | CFM #2 (infinite tool-call loops) | Hard cap at `maxIterations: 20`. In `CronHook.afterIteration()`, if `toolsUsed.length >= maxIterations`, inject a stop signal. | `AgentRunner` already enforces `maxIterations`. Additionally, the `CronHook` can detect repeated identical tool calls (>3 same tool+params in sequence) and inject a system message: "You have called the same tool 3+ times. Try a different approach." |
| **Destructive SQL prevention** | CFM #5 (permission escalation) | Intercept `db_sql_execute` tool calls. Before executing, parse the SQL statement. If it matches DDL/DML pattern (DROP, ALTER, TRUNCATE, DELETE, INSERT, UPDATE not in a SELECT-context), block the call and return error. | Implement in `cron-tool-catalog.ts` as a wrapper around the base `db_sql_execute`: parse SQL with a regex guard, return `"BLOCKED: Destructive operations are not permitted in cron tasks."` instead of executing. |
| **Scope enforcement** | CFM #5 (permission escalation) | Before each tool call, validate that the target instance/database is in the cron job's authorized scope list. | Add a `scopeCheck(toolName, params, allowedInstances)` guard in `CronHook.beforeExecuteTools()`. If the tool targets an instance not in the job's scope, skip execution and return `"SCOPE BLOCKED: Instance not authorized for this cron job."` |

### 6.2 Offline Flywheel (Batch Analysis)

Run as sampled batch processing. Feed results into system improvement loop -- do not block execution.

| Metric | What It Measures | Sampling | Collection | Action When Degraded |
|--------|-----------------|----------|------------|----------------------|
| **Task completion rate** | % of cron executions that complete with status `success` (vs. `failure`, `timeout`, `error`) | 100% of cron logs | Query `cron_job_logs` daily: `SELECT status, COUNT(*) FROM cron_job_logs WHERE executed_at > NOW() - INTERVAL 7 DAY GROUP BY status` | If completion rate < 90% over 7 days: investigate top-failing job IDs, review agent traces for common failure patterns. |
| **Execution trace quality** | % of logs with non-empty `tool_events`, non-null `stop_reason`, `summary` length > 50 chars | 100% of cron logs | Code check on insert: is the trace complete? If not, flag as `low_quality`. | If > 5% of logs are low-quality in a week: review system prompt, check for agent confusion, verify `CronHook` collection. |
| **Tool retry frequency** | How often tools fail and are retried, indicating instability | 100% of cron logs | Count `(toolName, error)` pairs in `tool_events` where `event.status === 'error'`. Group by tool. | If any single tool has > 20% error rate: investigate tool implementation or database connectivity. |
| **Cost per execution** | Token usage drift | 100% of cron logs | Sum `usage.input_tokens + usage.output_tokens` per execution. Track 7-day moving average. | If average cost per tick exceeds $1.00 (Claude Sonnet): review if tasks are excessively long, consider switching cheaper models for simple tasks. |
| **Actionability score trend** | D2 judge score moving average | 10% sampled production logs | LLM Judge (Section 5.2) scores sampled logs. Track 30-day moving average. | If average score drops below 3.0: review system prompt, verify agent is following analysis structure. |

## 7. Production Monitoring

### 7.1 Tracing Approach

No external tracing platform (Arize Phoenix, Langfuse, etc.) is deployed in this project. Adding one would be disproportionate complexity for an internal DBA tool. Instead:

**Primary trace store: `cron_job_logs` database table**

Each cron execution writes a complete trace record containing:

```sql
CREATE TABLE cron_job_logs (
  id INT AUTO_INCREMENT PRIMARY KEY,
  job_id INT NOT NULL,
  status ENUM('success', 'failure', 'partial', 'timeout', 'error') NOT NULL,
  result LONGTEXT,           -- final agent summary/report
  tools_used JSON,           -- array of tool names called
  tool_events JSON,          -- array of tool execution events with params and results
  usage JSON,                -- { input_tokens, output_tokens }
  stop_reason VARCHAR(50),   -- 'end_turn', 'max_turns', 'stop', 'length', 'error'
  executed_at DATETIME NOT NULL,
  duration_ms INT,           -- wall-clock execution time in milliseconds
  error_trace TEXT,          -- error details if status = error or timeout
  partial_trace LONGTEXT,    -- snapshots of partial results if timed out
  INDEX idx_job_id (job_id),
  INDEX idx_executed_at (executed_at),
  INDEX idx_status (status)
);
```

**Secondary trace: `.slide/sessions/` JSONL files**

Each cron execution's full message history is persisted by `SessionManager` as JSONL under `.slide/sessions/<sha256>.jsonl` keyed by `cron:<jobId>:<timestamp>`. Retain for 30 days, then rotate.

### 7.2 Key Metrics

| Metric | Source | Collection | Display |
|--------|--------|------------|---------|
| **Execution count** | `cron_job_logs` row count | Real-time (DB query) | Per-job dashboard, last 24h |
| **Status distribution** | `cron_job_logs.status` | Real-time | Pie chart by status, refresh 5min |
| **P50/P95/P99 duration** | `cron_job_logs.duration_ms` | Real-time | Histogram, per-job and global |
| **Tool error rate** | `tool_events` JSON parse | Batch, hourly | Per-tool error rate table |
| **Cost per execution** | `usage` JSON parse | Batch, daily | 7-day moving average trend line |
| **Active job count** | `cron_jobs` table WHERE enabled=1 | Real-time | Summary card |
| **Max tool calls per execution** | `tool_events` array length | Batch, hourly | Distribution chart |

### 7.3 Alert Thresholds

Targeted at operations engineers monitoring the platform. Alerts route to `#ops-alerts` channel (or equivalent).

| Condition | Threshold | Severity | Action |
|-----------|-----------|----------|--------|
| Job execution failure rate | > 10% over 1 hour | P2 | Investigate root cause: DB connectivity, tool errors, timeout pattern |
| Single job fails 3 consecutive ticks | 3 failures in a row for same job_id | P2 | Disable the job automatically, notify ops to review task description |
| Timeout rate increase | > 5% of executions timeout over 24h | P3 | Review if 5-min limit is sufficient, check for long-running queries |
| Destructive SQL attempt | Any blocked DDL/DML attempt | P1 | Immediate investigation -- potential security issue or prompt injection |
| Cost spike | > 2x 7-day average cost in a single day | P3 | Review which jobs are driving cost, consider model downgrade for simple tasks |

### 7.4 Sampling Strategy

Since this is an internal tool, store 100% of `cron_job_logs` entries (the data volume is low -- at most a few hundred entries/day for 13+ jobs). Sampling applies to:

| Data | Sampling Rate | Purpose |
|------|--------------|---------|
| Full `cron_job_logs` entry | 100% -- always store | Audit trail, traceability (requirement: non-repudiable log) |
| LLM Judge actionability scoring | 10% of logs, weighted toward `failure`/`timeout`/`partial` | Quality signal without per-execution cost |
| Human DBA review | 5% of logs, selected weekly, prioritize jobs with low actionability scores | Calibration for LLM Judge, edge case discovery |
| `.slide/sessions/` JSONL | 100% during execution, rotate after 30 days | Full trace for incident investigation |

**Weighted sampling for LLM Judge:**
- All `status = 'failure'` or `status = 'timeout'` logs are scored (100%)
- 10% of `status = 'success'` logs are scored (random sample)
- If a job's actionability scores drop below 3.0 for 3 consecutive samples, temporarily increase its sampling rate to 50%
