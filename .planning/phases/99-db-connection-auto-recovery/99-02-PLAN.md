**现有流程分析：**

`collectInstanceMetrics(instance)` 方法（第 182-197 行）：
```
try {
  const metrics = await databaseService.getRealtimeMetrics(instance.id);
  if (metrics) {
    await metricsDatabaseService.recordMetrics({...});
    await this.checkAlerts(instance.id, metrics);
  }
  await this.updateHealthStatusFromCheck(instance.id);
} catch (error) {
  console.error(`采集实例 ${instance.name} 指标失败:`, error);
}
```

问题：当数据库断开时，`getRealtimeMetrics` 通过 Plan 01 的 `_withAutoReconnect` 会尝试重连。但如果重连也失败（DB 还没恢复），返回 null（不是 throw）。此时 `metrics` 为 null，`if (metrics)` 不执行，然后进入 `updateHealthStatusFromCheck`。

但 `updateHealthStatusFromCheck`（第 201 行）调用 `databaseService.checkHealth()`，也需要连接。如果 DB 仍然不可用，`checkHealth` 返回 critical 状态。

真正的缺口：当数据库从断连恢复到可用时，需要 **主动触发** 重连尝试，而不是被动等下一次 `_tick` 循环（可能要 10 秒或更久）。

**目标：** 在 catch 块中（getRealtimeMetrics 抛出异常的情况），或者 `getRealtimeMetrics` 返回 null 但实例状态为 active 时，主动调用 `databaseService.checkConnectionAlive()` 和 `databaseService.reconnect()` 尝试恢复。

注意：Plan 01 之后，`getRealtimeMetrics` 内部已经会通过 `_withAutoReconnect` 尝试重连。但 `_withAutoReconnect` 只重试一次。如果失败了，monitor-collector 应该在下一个心跳周期再次尝试——这是通过正常的心跳循环自然完成的。

但 `updateHealthStatusFromCheck` 还有一个独立问题：当数据库断开重连后，`updateHealthStatusFromCheck` 应该把健康状态恢复为 `healthy`。

实际上更关键的问题是：Plan 01 在 `_withAutoReconnect` 的重连失败后直接返回 null，但 monitor-collector 拿到 null 后什么也不做。DB 恢复后下一次 `_tick` 又会尝试 `getRealtimeMetrics`，此时 `_withAutoReconnect` 会再次尝试重连。所以心跳循环本身就是重试机制。

真正的修复点是：**`collectInstanceMetrics` 中 `getRealtimeMetrics` 返回 null 时，说明重连失败，应该更新 instance health status 为 critical。**

但现有代码中 `updateHealthStatusFromCheck` 已经做了这件事（`checkHealth` 返回 null 时会在 catch 里写 critical）。所以现有的心跳循环已经能自动修复——只是一次 tick 的时间差（10s）。

**重新审视 Scope：**

scope 描述的 Plan 02 要求：
1. 在 `collectInstanceMetrics()` catch 块：调用 `databaseService.checkConnectionAlive()` 
2. 如果健康检查失败：调用 `databaseService.reconnect()`
3. 重连成功：调用 `instanceDatabaseService.updateHealthStatus(id, 100, 'healthy')`
4. 重连失败：保持错误状态

这个行为是在**现有心跳循环之外**增加一个主动重连层。不同于 `_withAutoReconnect` 的"查询时被动重连"，这是"采集失败后主动重连"。

**边角情况：** 实例在重连过程中被删除 → `databaseService.reconnect()` 会返回 false（因为 `connections` 中已经不存在该 id 的 entry），所以第二步 `updateHealthStatus` 不会执行。

    ```
    private async collectInstanceMetrics(instance: any) {
      try {
        const metrics = await databaseService.getRealtimeMetrics(instance.id);
        if (metrics) {
          await metricsDatabaseService.recordMetrics({
            ...metrics,
            instance_id: instance.id,
          });
          await this.checkAlerts(instance.id, metrics);
          console.log(`📊 [${instance.name}] CPU: ${metrics.cpu_usage}%, Memory: ${metrics.memory_usage}%, Connections: ${metrics.connections}`);
        } else {
          // getRealtimeMetrics 返回 null —— 连接不可用或重连失败
          // 尝试主动重连
          console.log(`⚠️ [${instance.name}] 指标采集返回 null，尝试主动重连...`);
          const reconnected = await databaseService.reconnect(instance.id);
          if (reconnected) {
            console.log(`✅ [${instance.name}] 主动重连成功，更新健康状态`);
            // 重连后立即标记为 healthy（不等待下一个 tick）
            // 注意：更新 health_status 为 healthy 但保留 db_version 和 data_size_gb 不修改
            await instanceDatabaseService.updateHealthStatus(instance.id, 100, 'healthy');
            // 尝试重新采集一次
            const retryMetrics = await databaseService.getRealtimeMetrics(instance.id);
            if (retryMetrics) {
              await metricsDatabaseService.recordMetrics({
                ...retryMetrics,
                instance_id: instance.id,
              });
              await this.checkAlerts(instance.id, retryMetrics);
              console.log(`📊 [${instance.name}] 重连后采集成功：CPU: ${retryMetrics.cpu_usage}%`);
            }
          } else {
            console.error(`❌ [${instance.name}] 主动重连失败`);
          }
        }
        // 无论 metrics 是否成功，都执行健康状态检查（通过 checkHealth 算法）
        await this.updateHealthStatusFromCheck(instance.id);
      } catch (error) {
        console.error(`采集实例 ${instance.name} 指标失败:`, error);
        // 异常路径：尝试主动重连恢复
        let recoverySucceeded = false;
        try {
          const alive = await databaseService.checkConnectionAlive(instance.id);
          if (!alive) {
            console.log(`⚠️ [${instance.name}] 采集异常 + 连接不可用，尝试重连...`);
            const reconnected = await databaseService.reconnect(instance.id);
            if (reconnected) {
              console.log(`✅ [${instance.name}] 异常后重连成功，更新健康状态`);
              await instanceDatabaseService.updateHealthStatus(instance.id, 100, 'healthy');
              recoverySucceeded = true;
            }
          } else {
            // 连接还活着，采集失败不是连接问题，不标记 critical
            recoverySucceeded = true;
          }
        } catch (recoveryError) {
          console.error(`❌ [${instance.name}] 恢复尝试失败:`, recoveryError);
        }
        // 仅在重连失败时才标记 critical
        if (!recoverySucceeded) {
          await instanceDatabaseService.updateHealthStatus(instance.id, 0, 'critical');
        }
      }
    }
    ```

    - `instanceDatabaseService.updateHealthStatus(id, 100, 'healthy')` 重连成功后恢复状态，让 UI 实时同步
    - 异常的 catch 块末尾调用 `updateHealthStatus(id, 0, 'critical')` 确保实例状态正确
