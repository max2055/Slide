<objective>
Add JWT authentication to the DirectAdapter WebSocket transport.

**Purpose:** The frontend DirectGatewayClient sends `{type:'auth', token}` as the first message after WS open. The backend must verify this JWT before processing any chat messages. Auth failure causes an immediate `close(4001, 'Unauthorized')` — the frontend stops reconnecting on this code.

**Output:**
- `direct-adapter.ts` — Auth frame handler (verify JWT, reject unauthenticated messages)
</objective>

  <action>
    Add JWT auth handling to the WebSocket onmessage handler in direct-adapter.ts. The handler currently dispatches on msg.type === 'chat.send' / 'chat.history'. Insert an auth check BEFORE the existing switch:

    1. Import `jwt` at top of file: `import jwt from 'jsonwebtoken';`
    2. Create a private `_authUserId` property on the WebSocket instance to track authenticated state.
    3. In the onmessage handler, before the `switch (msg.type)`:
       - If `msg.type === 'auth'`:
         - Read `msg.token` as string
         - Call `jwt.verify(token, process.env.JWT_SECRET_KEY)` — the JWT_SECRET_KEY env var is already set from server.ts::L78
         - On success: set `(ws as any)._authUserId = (decoded as any).userId;` then send `ws.send(JSON.stringify({ type: 'auth_ok' }))` and return
         - On error: call `ws.close(4001, 'Unauthorized')` and return
       - If `(ws as any)._authUserId` is falsy and msg.type !== 'auth' (catch unauthenticated non-auth messages):
         - Send `ws.send(JSON.stringify({ type: 'error', error: 'Authentication required' }))` and return
    4. Keep the existing chat.send switch as-is — no changes to the chat flow.
    5. The error handler for invalid JSON in the try/catch is unchanged.

    Design decisions per D-09/D-10/D-11:
    - JWT is only verified at connection time (D-11). Once authenticated, the connection stays valid.
    - Auth failure = close(4001) — no retry, no fallback (D-10).
    - No guest mode: every connection must authenticate before sending any message.
  </action>
