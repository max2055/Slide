# Slide 项目路线图 V2

> 基于 2026-04-23 讨论更新，补充数据库运维管理功能体系化规划。

## 全局约束

| 约束 | 说明 |
|------|------|
| 只读账号 | 系统纳管数据库仅使用只读账号，不支持 DDL 执行 |
| 无 OS 访问 | 没有操作系统级账号，无法采集 OS 日志或执行系统命令 |
| LLM 可用 | SQL 审核和 AI 分析可使用 LLM 能力 |
| 现有基础 | M1 数据采集、告警引擎、报表生成已完成 |

## M4: 功能完善

> 目标：补全日常运维最高频的基础功能——通知推送、可视化、表结构管理、索引采集。
> 前提：复用现有告警规则表，不做引擎升级。

### Phase 4.1 - 告警通知推送

**目标：** 告警触发后能自动推送给管理员，形成闭环。

**通知渠道：**
- 钉钉 Webhook / 机器人
- 企业微信 Webhook
- 飞书 Webhook
- 通用 Webhook

**功能范围：**

| 功能 | 说明 |
|------|------|
| 通知渠道管理 API | CRUD 渠道配置（名称、类型、URL、密钥） |
| 通知模板 | 按告警等级（info/warning/error/critical）定义消息模板 |
| 通知路由 | 告警触发时匹配渠道规则（按等级/类型/实例过滤） |
| 通知记录 | 记录发送结果（成功/失败、时间、错误信息） |
| 测试发送 | API 支持测试渠道连通性 |

**复用现有数据：**
- `alert_rules` 表 — 规则已包含 severity
- `alerts` 表 — 告警事件已有完整信息
- 告警引擎 — 每 60 秒评估，触发时同时推送通知

**技术实现：**
- 方案 B：独立通知服务（NotificationService），轮询 `alerts` 表的新增记录，匹配通知渠道后异步发送
- 通知发送与告警引擎解耦，避免通知失败影响告警评估
- 失败重试机制（最多 3 次，间隔递增）
- 通知记录写入 `notification_logs` 表

**实现细节：**
- NotificationService 每 10 秒轮询一次 alerts 表（status = unread 且未推送过的）
- 按 alert 的 severity 和 tag 匹配通知渠道的路由规则
- 通知模板支持变量替换：`{alert_title}`, `{alert_level}`, `{instance_name}`, `{metric_value}`, `{threshold}`

### Phase 4.2 - 指标可视化

**目标：** 用户能看到指标的历史趋势，而不是只看当前快照。

**可视化内容：**
- CPU 使用率趋势图
- 内存使用率趋势图
- QPS / TPS 趋势图
- 连接数趋势图
- 慢查询数趋势图
- 磁盘使用率趋势图
- 自定义时间范围（1h / 6h / 24h / 7d）

**技术方案：**
- 前端集成 ECharts 或 Chart.js
- 后端 `metrics_history` 表已有历史数据
- 新增 API：`GET /api/database/instances/:id/metrics/history?period=1h&interval=5m`
- 支持多实例对比（可选）

**数据来源：**
- `metrics_history` 表 — 每 30 秒采集一次
- `metricsDatabaseService.getHistoricalMetrics()` — 已有历史查询方法

---

### Phase 4.3 - 表结构管理

**目标：** 记录数据库表结构的变更历史，支持 Schema 快照对比。

**功能范围：**

| 功能 | 说明 |
|------|------|
| Schema 快照采集 | 从 information_schema 采集表结构元数据（表名、列、类型、注释） |
| 变更检测 | 对比上次快照，识别新增/修改/删除的表和列 |
| 变更历史 | 展示 DDL 变更记录列表（时间、类型、对象名称） |
| 快照对比 | 选择两个时间点，展示表结构差异 |
| 表详情 | 展示单表的完整结构信息 |

**技术要点：**
- 从 `information_schema.TABLES` 和 `information_schema.COLUMNS` 采集
- 快照存储到 `schema_snapshots` 表（JSON 格式存储完整结构）
- 变更对比算法：基于表名/列名的 diff 比对
- 定时采集（每 1 小时）+ 手动触发
- **不做**：DDL 执行、Schema 版本回滚

---

### Phase 4.4 - 索引管理

**目标：** 采集并展示索引信息，为 M5 的 AI 分析提供数据基础。

**功能范围：**

| 功能 | 说明 |
|------|------|
| 索引采集 | 从 information_schema.STATISTICS 采集索引信息 |
| 索引列表 | 按实例/表展示所有索引（名称、类型、列、基数） |
| 冗余索引检测 | 基于规则检测冗余/重复索引（如 (a,b) 和 (a) 的关系） |
| 未使用索引 | 标记长期未使用的索引（基于 performance_schema） |
| 索引统计信息 | 索引大小、行数、选择率 |

**技术要点：**
- 采集 SQL：`SELECT * FROM information_schema.STATISTICS WHERE table_schema = ?`
- 冗余检测算法：基于索引前缀匹配（(a,b,c) 包含 (a,b) 包含 (a)）
- 未使用索引检测依赖 performance_schema（MySQL 5.7+）
- **不做**：索引创建/删除操作（只读模式）

## M5: AI 增强

> 目标：在运维操作场景中嵌入 AI 分析能力，由 agent 主动采集信息并给出分析结论。
> 触发方式：事件触发 + 手动触发。

### Phase 5.1 - TopSQL AI 分析

**目标：** 慢查询列表中的每条 SQL，AI 自动分析并给出优化建议。

**分析流程：**
1. 获取慢查询 SQL 文本和执行统计（耗时、扫描行数）
2. 获取 EXPLAIN 执行计划
3. 获取相关表的索引信息（M4.4 采集的数据）
4. 获取相关表的列信息和数据量
5. 调用 LLM 综合分析，生成优化建议

**输出内容：**
- 执行计划解读（全表扫描/索引扫描/临时表/文件排序）
- 索引优化建议（推荐新增索引、现有索引未命中原因）
- SQL 改写建议（如果适用）
- 风险等级（P0/P1/P2）

**触发方式：**
- 手动：用户点击某条慢查询的 "AI 分析" 按钮
- 事件：新采集的慢查询如果超过阈值（如执行时间 > 10s），自动分析

---

### Phase 5.2 - 告警根因分析

**目标：** 告警触发后，AI agent 自动拉取相关数据，分析根因。

**分析流程：**
1. 告警事件触发（如 CPU 过高）
2. Agent 自动采集：该实例最近 5 分钟的指标趋势、活跃会话列表、慢查询列表、锁等待信息、最近 Schema 变更
3. 将所有信息打包发送给 LLM
4. LLM 生成根因分析报告

**输出内容：**
- 告警概述（什么指标、什么阈值、何时触发）
- 根因分析（可能导致该告警的原因排序）
- 相关证据（支撑分析的具体数据）
- 建议操作（逐步修复指引）
- 关联告警（是否有其他告警由同一根因引起）

**触发方式：**
- 事件触发：告警等级为 warning/error/critical 时，自动触发根因分析（info 级别不触发）
- 手动触发：用户在告警列表点击 "AI 分析"，支持重新触发（对自动结果不满意时可手动再次分析）

**技术要点：**
- 异步执行，不阻塞告警引擎
- 分析结果写入 `alert_diagnoses` 表
- 前端页面展开显示分析结果（非弹窗）
- 支持轮询获取分析进度（每 3 秒检查一次）
- 同一告警的自动分析结果缓存，手动重新触发时忽略缓存重新分析

---

### Phase 5.3 - 故障自动诊断

**目标：** 实例健康评分下降时，AI agent 执行完整诊断流程。

**触发条件：**
- 健康评分从 healthy 下降到 warning（< 80）
- 健康评分从 warning 下降到 critical（< 50）
- 手动触发用户点击 "一键诊断"

**诊断流程：**
1. 获取当前健康评分和各维度得分
2. 获取最近指标数据（CPU/内存/连接/IO）
3. 获取活跃会话（是否有长事务、锁等待、阻塞）
4. 获取慢查询列表
5. 获取最近告警记录
6. 获取索引状态（是否有损坏索引）
7. 获取复制状态（如果是主从架构）
8. LLM 综合分析生成诊断报告

**输出内容：**
- 整体健康评估（当前状态、趋势判断）
- 问题列表（按严重程度排序）
- 根因推断
- 修复建议（含优先级和风险评估）

---

### Phase 5.4 - 前端 AI 分析入口

**目标：** 在前端页面中嵌入 AI 分析功能的交互入口。

**AI 功能定位：**
- AI 功能分为 2 大块：**对话**（聊天中通过 tool-use 调用工具）和 **AI 分析**（在运维场景中由 agent 主动采集信息分析）
- AI 分析统一采用事件触发 + 手动触发，产生结果后保存在数据库，前端页面展开显示
- 用户对自动结果不满意时，可手动再次触发分析

**交互设计：**

| 页面 | 入口 | 展示 |
|------|------|------|
| TopSQL 列表 | 每条慢查询旁的 "AI 分析" 按钮 | 页面内展开分析结果面板 |
| 告警列表 | 每条告警旁的 "根因分析" 按钮 | 页面内展开根因分析报告 |
| 实例详情 | "一键诊断" 按钮 | 页面内弹出诊断结果面板 |
| 告警详情 | 自动展示关联的根因分析 | 分析报告卡片（页面内） |

**技术要点：**
- AI 分析是异步任务，前端通过轮询获取结果（每 3 秒检查一次）
- 分析结果缓存（避免重复调用 LLM），手动重新触发时忽略缓存
- 分析失败时展示友好的降级提示
- 分析成本考虑：控制 LLM 调用频率，同一资源 N 分钟内不重复分析

---

### Phase 5.5 - 容量预测

**目标：** 基于历史数据，用线性回归预测存储和连接数的使用趋势。

**预测内容：**
- 磁盘存储容量预测（何时用满）
- 连接数增长趋势（是否接近上限）

**技术方案：**
- 简单线性回归：`y = mx + b`
- 输入数据：`metrics_history` 表的 `disk_usage`、`connections` 历史数据
- 预测未来 7 天 / 30 天 / 90 天的趋势
- 在前端展示预测曲线（虚线表示预测区间）

**展示方式：**
- 容量趋势图表（实线 = 历史数据，虚线 = 预测曲线）
- "预计 X 天后存储用满" 的文字提示
- 超过 80% 时触发预警（结合告警规则）

## M6: 体系化重构

> 目标：建立指标体系和告警体系，让数据采集和告警管理有统一的规范。

### Phase 6.1 - 指标体系定义与管理

**目标：** 统一指标定义，解决当前指标散落在代码中、无统一管理的问题。

**指标注册：**
- 代码里定义指标 schema（名称、单位、类型、适用数据库、采集方法、采集频率、开销等级）
- 配置表 `metric_definitions` 管理指标元数据
- 告警规则引用的 metric_name 必须来自已注册指标

**指标分类：**
- 按数据库类型：MySQL、PostgreSQL、Oracle、达梦、Redis、Elasticsearch
- 按维度：性能、资源、连接、查询、缓存、复制

**启用/禁用：**
- 高开销指标默认禁用（如详细锁分析）
- 按实例配置指标采集策略（生产实例全量采集，测试实例只采集核心指标）

**技术实现：**
- 创建 `metric_definitions` 配置表
- 代码中定义指标注册器（IndicatorRegistry 类）
- 采集器根据注册表的 enabled 字段决定是否采集

---

### Phase 6.2 - 告警规则引擎升级

**目标：** 从简单阈值告警升级为多层级的告警规则引擎。

**升级内容：**

| 功能 | 说明 |
|------|------|
| 多层级规则 | 指标定义 → 告警规则 → 告警事件，三层解耦 |
| 基线阈值 | 支持动态基线（过去 N 天同期均值 ± σ） |
| 静默期 | 同一规则同一实例 N 分钟内不重复告警 |
| 告警升级 | Warning 持续 N 分钟自动升级为 Error |
| 环比告警 | 支持相对变化（QPS 突降 50%） |
| 多条件规则 | 支持 AND/OR 组合条件（CPU > 80% AND 连接数 > 100） |

**向后兼容：**
- 现有 `alert_rules` 表扩展字段，不新建表
- 现有规则自动迁移为新格式（固定阈值模式）

---

### Phase 6.3 - 告警事件管理优化

**目标：** 告警事件管理更精细化。

**升级内容：**

| 功能 | 说明 |
|------|------|
| 告警分组 | 同一根因的多个告警合并为一个事件组 |
| 告警状态流转 | 未确认 → 已确认 → 处理中 → 已解决 → 已关闭 |
| 告警分配 | 告警分配给特定处理人 |
| 告警标签 | 为告警添加标签（如 "生产环境"、"主库"） |
| 告警统计 | 按时间/实例/等级的告警分布统计 |

---

## M7: 扩展功能

> 目标：扩展更高级的数据库运维管理能力。

### Phase 7.1 - SQL 审核（LLM 驱动）

**目标：** 通过 LLM 分析 SQL 语句的正确性、性能和风险。

**功能范围：**

| 功能 | 说明 |
|------|------|
| 语法检查 | 检测 SQL 语法错误 |
| 性能风险 | 评估潜在性能问题（全表扫描、缺少 WHERE 等） |
| DDL 风险 | 评估 DDL 变更的风险（大表加列、删除列等） |
| 规范检查 | 是否符合团队 SQL 编码规范 |
| 审核日志 | 记录所有审核请求和结果 |

**技术方案：**
- 使用 LLM 判断 SQL 风险等级和建议
- 定义审核提示模板（system prompt）
- 审核结果缓存，避免重复请求

---

### Phase 7.2 - 用户权限管理

**目标：** 管理数据库层的用户和权限。

**功能范围：**

| 功能 | 说明 |
|------|------|
| 用户列表 | 查看数据库用户列表 |
| 权限查看 | 查看用户的权限信息（SHOW GRANTS） |
| 权限变更追踪 | 记录权限变更记录（只读展示） |

**注意：** 只读模式限制，部分操作可能受限。

---

### Phase 7.3 - 日志分析

**目标：** 分析数据库错误日志。

**功能范围：**

| 功能 | 说明 |
|------|------|
| 错误日志采集 | 从数据库系统表/性能视图采集错误日志 |
| 错误统计 | 按错误类型统计出现次数 |
| 错误趋势 | 错误数量的时间趋势 |

**限制：** 无法采集 OS 层日志（无 OS 访问权限）。

## 阶段依赖关系

```
M4.1 (告警通知推送) ──┐
                      ├── 无强依赖，可独立实现
M4.2 (指标可视化) ────┤
                      │
M4.3 (表结构管理) ──┐ │
                   ├── 为 M5.1/M5.2 提供数据基础
M4.4 (索引采集) ────┘ │
                      │
M5.1 (TopSQL AI) ─────┤ 依赖 M4.4
M5.2 (告警根因分析) ──┤ 依赖 M4.1 (通知) + M4.3/M4.4
M5.3 (故障诊断) ──────┤
M5.4 (前端 AI 入口) ──┤ 依赖 M5.1-M5.3
M5.5 (容量预测) ──────┤ 依赖 M4.2 (历史数据)
                      │
M6.1 (指标体系) ──────┤
M6.2 (告警规则升级) ──┤ 依赖 M4.1 + M6.1
M6.3 (告警事件优化) ──┘ 依赖 M6.2
```
